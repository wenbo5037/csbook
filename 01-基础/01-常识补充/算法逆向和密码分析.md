# 算法逆向与密码分析

## 绪论：计算复杂性与安全测试中的密码学壁垒

现代密码学的设计严格遵循**柯克霍夫原则（Kerckhoffs's Principle）**：系统的安全性不应依赖于算法的保密，而应仅依赖于密钥的保密。在评估任何加密系统时，我们基于**计算复杂性理论**，假设攻击者拥有图灵完备的计算能力，并试图在多项式时间（Polynomial Time, P）内破解系统。密码学安全的核心在于构造**单向陷门函数（One-way Trapdoor Functions）**。

在真实的安全测试（黑盒/白盒/灰盒）与红蓝对抗中，密码学往往是横亘在漏洞探针与后渗透之间的最大壁垒。若无法识别传输层的编码与加密，漏洞Payload将无法进入业务逻辑（漏洞探针失效）；若无法理解回显数据的加密格式，盲注与盲打将失去判断依据（数据分析失效）；若无法破解密码存储机制或系统哈希，即使获取了数据库或Shell，也无法完成横向移动与权限维持（后渗透失效）。

克劳德·香农（Claude Shannon）奠定了现代密码学的两大基石：**混淆（Confusion）**与**扩散（Diffusion）**。混淆旨在使密文与密钥之间的统计关系极度复杂化（如AES的S盒非线性替换）；扩散旨在使明文的微小变化迅速散布到整个密文中（如AES的MixColumns）。任何未能同时满足这两点的密码系统，都将在差分密码分析或线性密码分析下原形毕露。然而，数学模型的完美无法掩盖工程实现的脆弱。本文将全面解剖这些从理论到工程的致命裂缝。

## 第一章：数据编码、传输格式与协议层安全分析

在Web与APP安全测试中，数据传输的编码与格式是漏洞挖掘的第一道关卡。许多测试者在面对加密或特殊编码的参数时，直接提交明文Payload，导致请求被WAF拦截或被后端业务逻辑直接丢弃。理解传输层的编码与格式，是确保Payload成功“入洞”的前提。

### 1.1 编码与加密的本质区别及实战影响

**编码（Encoding）** 的目的是为了数据在不同系统间安全传输，不涉及密钥，具备100%的可逆性。常见的编码包括 Base64、URL-Encoding、HTML-Encoding、Unicode 等。
**加密（Encryption）** 的目的是为了数据保密性，依赖密钥和算法，具备计算复杂性壁垒。

**实战影响：漏洞探针的失效与重构**
假设目标URL为 `index.php?id=Mq==`。通过 Base64 解码可知 `Mq==` 对应明文 `1`。若测试者直接提交 `id=1' OR 1=1--`，后端在接收到该字符串后进行 Base64 解码，会得到乱码或报错，导致 SQL 注入 Payload 根本无法进入数据库查询逻辑。
**正确做法**：必须将 Payload `1' OR 1=1--` 进行 Base64 编码为 `MScgT1IgMT0xLS0=`，再提交 `id=MScgT1IgMT0xLS0=`。后端解码后还原出恶意 SQL 语句，从而触发漏洞。这要求测试者必须具备“同态构造”思维，即 Payload 必须经过与合法数据相同的编码/加密管道。

### 1.2 传输数据格式深度解析与衍生漏洞

现代应用的数据传输格式主要分为 Form-urlencoded、JSON、XML 以及二进制的 Protobuf。不同格式不仅影响数据的序列化方式，更直接决定了攻击面的大小。

#### 1.2.1 JSON 格式与类型混淆/反序列化
JSON（JavaScript Object Notation）是目前 API 交互的主流格式。其严格的键值对结构（双引号包裹键名）和类型支持（字符串、数字、布尔、数组、对象）为测试带来了新的挑战。
- **类型混淆（Type Juggling）**：在 PHP 等弱类型语言中，若后端期望接收字符串 `{"id": "1"}`，但攻击者提交数字 `{"id": 1}` 或数组 `{"id": ["1"]}`，可能导致 `md5($id)` 报错或 `strcmp()` 绕过。
- **Mass Assignment（批量赋值）**：在 Ruby on Rails 或 Spring Boot 中，若后端直接将 JSON 对象映射到数据库模型，攻击者可通过在 JSON 中注入 `{"role": "admin"}` 或 `{"is_vip": true}` 实现越权。
- **JSON 反序列化**：在 Java (Fastjson/Jackson) 或 .NET (Json.NET) 中，若开启了多态类型处理（如 `@type` 或 `TypeNameHandling`），攻击者可构造恶意 JSON 触发 RCE。

#### 1.2.2 XML 格式与 XXE 注入
XML（eXtensible Markup Language）因其冗长和解析复杂，逐渐被 JSON 取代，但在企业级应用（如 SOAP Web Services、SAML 单点登录）中依然广泛存在。XML 的 DTD（文档类型定义）特性使其天然容易受到 **XXE（XML External Entity）** 攻击。
**攻击原理**：XML 解析器默认允许加载外部实体。攻击者可通过构造恶意 DTD，读取服务器本地文件、执行 SSRF 或导致 DoS（Billion Laughs 攻击）。
```xml
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE foo [
  <!ENTITY xxe SYSTEM "file:///etc/passwd">
]>
<user><username>&xxe;</username></user>
```
**防御与绕过**：现代解析器默认禁用外部实体。若后端仅禁用了 `file://` 协议，攻击者可尝试 `php://filter`、`dict://` 或通过 Out-of-Band (OOB) 技术将数据外带至攻击者服务器。

#### 1.2.3 Protobuf 与二进制协议逆向
在高性能 APP 和微服务架构中，Google 的 Protocol Buffers (Protobuf) 被广泛用于替代 JSON。Protobuf 将数据序列化为紧凑的二进制流，抓包时表现为不可读的高熵乱码。
**逆向分析**：Protobuf 的二进制格式包含 Field Number 和 Wire Type。通过工具（如 `protoc --decode_raw`）可将其还原为键值对结构，但会丢失字段名称。若要完全还原，需从 APK 或后端二进制中提取 `.proto` 定义文件，或使用 `Blackbox Protobuf` 等 Burp 插件进行动态 Hook 与篡改。

### 1.3 抓包与流量分析实战：突破 TLS 与证书校验

在 APP 和桌面客户端的安全测试中，流量拦截是第一步。现代应用普遍采用 TLS 1.2/1.3，并频繁使用 **SSL Pinning（证书绑定）** 技术，导致 Burp Suite 或 Charles 等常规中间人（MITM）工具失效。

#### 1.3.1 SSL Pinning 原理与绕过
SSL Pinning 强制客户端只信任预置在应用内部的特定证书（或公钥哈希），从而防御 MITM 攻击。
**绕过方案**：
1. **Frida 动态插桩**：使用 `objection` 或编写 Frida 脚本，Hook 底层的 SSL 校验函数（如 Android 的 `X509TrustManager.checkServerTrusted`，iOS 的 `NSURLSessionDelegate` 或 `AFNetworking` 的校验逻辑），强制其返回 `true`。
2. **网络层 Hook**：直接 Hook `SSL_read` 和 `SSL_write`（OpenSSL/BoringSSL），在数据被加密前和解密后直接抓取明文，彻底 bypass TLS 层。
3. **重打包与 Patch**：使用 `apktool` 反编译 APK，利用 `JustTrustMe` 或手动修改 Smali 代码移除证书校验逻辑，重新签名后安装。

#### 1.3.2 双向认证 (mTLS) 与流量解密
在金融、医疗等高安全场景中，服务器不仅向客户端证明身份，还要求客户端提供证书（mTLS）。
**应对策略**：若客户端证书硬编码在 APP 或 P12 文件中，需通过逆向提取证书及密码。若证书通过服务端动态下发并存储在系统安全区域（如 Android Keystore / iOS Secure Enclave），则无法直接提取。此时必须使用 **SSLKEYLOGFILE** 技术：Hook 客户端底层的 SSL 库，提取握手阶段生成的 Pre-Master Secret，导出为 Wireshark 可识别的密钥日志文件，从而在旁路解密流量。

### 1.4 回显数据分析与盲注/盲打中的加密回显处理

在 SQL 盲注或 XSS 盲打中，攻击者依赖服务器的回显（如页面长度、HTTP 状态码、特定错误信息）来判断 Payload 是否生效。若后端对返回数据进行了全局加密（如 AES-CBC），攻击者将无法直接观察回显差异。
**分析策略**：
1. **密文差异分析**：即使数据被加密，不同的明文（如“查询成功”与“查询失败”）经过 AES-CBC 加密后，其密文的 Base64 长度或特定 Block 的内容必然不同。通过统计密文特征，可建立“密文指纹”库，实现盲注判断。
2. **Padding Oracle 利用**：若后端在解密失败时返回 500 错误，在解密成功但业务失败时返回 200 错误，攻击者可利用 Padding Oracle 攻击，逐字节解密历史回显密文，甚至加密任意伪造的回显数据。

## 第二章：单向散列函数、密码存储机制与碰撞分析

哈希函数 $H: \{0,1\}^* \rightarrow \{0,1\}^n$ 必须满足抗原像、抗第二原像和抗强碰撞。在密码存储与完整性校验中，哈希算法的应用极为广泛，但其工程实现的缺陷往往是系统被攻破的突破口。

### 2.1 密码存储的演进与实战破解

#### 2.1.1 裸 MD5 与彩虹表碰撞
早期 CMS（如 ZZCMS 老版本）直接将用户密码进行 MD5 哈希后存入数据库。MD5 输出固定 128 位（32个十六进制字符）。由于其计算速度极快，现代 GPU 每秒可进行数百亿次 MD5 运算。
**破解手段**：对于裸 MD5，攻击者无需进行实时暴力破解，只需查询预先计算好的**彩虹表（Rainbow Table）** 或使用在线 API（如 cmd5.com）。若密码为弱口令（如 `123456`），瞬间即可还原。

#### 2.1.2 MD5 加盐 (Salt) 与 Discuz 3.2 机制
为了防御彩虹表，现代系统引入了 Salt。以 Discuz 3.2 为例，其密码存储格式为 `md5(md5($password) . $salt)`。Salt 通常是随机生成的 6 位字符串，与哈希值一同存储在数据库中。
**实战破解**：
当攻击者通过 SQL 注入或备份泄露获取了数据库后，必须同时提取 `password` 和 `salt` 字段。由于 Salt 的引入，预计算的彩虹表彻底失效。攻击者必须使用 Hashcat 等工具，指定 `--hash-type 2611` (vBulletin/Discuz MD5+Salt)，结合自定义字典进行实时暴力破解。若密码复杂度较高（包含大小写、数字、特殊符号且长度>10），破解成功率将呈指数级下降。

#### 2.1.3 复杂加密与 Discuz 3.5 机制
在 Discuz 3.5 及更现代的系统（如 WordPress 的 phpass、Django 的 PBKDF2）中，密码存储不再使用简单的 MD5，而是采用 **bcrypt、scrypt 或 Argon2** 等**慢哈希（Memory-Hard Functions）** 算法。
**机制分析**：以 bcrypt 为例，其密文格式为 `$2a$10$N9qo8uLOickgx2ZMRZoMye...`。`$2a$` 标识算法，`10$` 标识 cost 因子（迭代次数为 $2^{10}$），后续 22 个字符为 Base64 编码的 128 位 Salt，最后 31 个字符为哈希值。
**实战影响**：慢哈希算法在设计上故意消耗大量 CPU 和内存资源。即使攻击者获取了密文和 Salt，使用顶级 GPU 集群每秒也只能尝试数千次。对于复杂密码，破解在物理时间上变得不可行。此时，安全测试的重点应从“破解哈希”转向“寻找逻辑漏洞”（如密码重置逻辑缺陷、Session 劫持）。

### 2.2 长度扩展攻击 (Length Extension Attack) 与 HMAC

MD5、SHA-1、SHA-256 均基于 **Merkle-Damgård (MD) 迭代结构**。该结构将消息分块，通过压缩函数迭代更新内部状态。最终的哈希值直接暴露了算法的最终内部状态。

**攻击原理**：
假设某 API 使用 `signature = SHA256(secret_key + user_data)` 进行防篡改校验。攻击者截获了 `user_data` 和 `signature`。
1. 攻击者将 `signature` 还原为 SHA256 的 8 个 32 位内部寄存器状态。
2. 构造附加 Payload（如 `&role=admin`）。
3. 使用工具（如 `HashPump`），以截获的 `signature` 作为初始状态，以 `len(secret_key) + len(user_data) + padding` 作为消息长度，计算附加 Payload 的新哈希。
4. 构造伪造请求：`user_data + SHA256_Padding + 附加Payload`，并附带新计算的签名。后端校验时，内部状态流转与攻击者完全一致，校验通过。

**防御与 HMAC**：
彻底免疫长度扩展攻击的方案是使用 **HMAC** 或 **SHA-3 (Keccak)**。HMAC 通过双重哈希结构 $H((K' \oplus opad) || H((K' \oplus ipad) || m))$，使得外层哈希的输入被内层哈希保护，攻击者无法获知中间状态。

### 2.3 系统级凭证提取与哈希破解

在获取系统 Shell 或数据库权限后，提取并破解系统级凭证是后渗透（Post-Exploitation）的核心环节。

#### 2.3.1 Windows NTLM 哈希与 Pass-the-Hash
Windows 本地账户密码存储在 SAM 数据库中，域账户密码存储在 NTDS.dit 中。其哈希算法为 **NTLM (NT LAN Manager)**。
**NTLM 结构**：NTLMv1/v2 并非基于 Salt 的慢哈希，而是将 Unicode 密码分为两个 7 字节块，分别作为 DES 密钥对固定字符串进行加密，然后拼接。这种设计极其脆弱。
**实战利用**：
1. **提取**：使用 `mimikatz`、`secretsdump.py` 或 `reg save` 提取 SAM/NTDS 哈希。
2. **破解**：由于 NTLM 无 Salt 且计算极快，使用 Hashcat (`-m 1000`) 可在数分钟内破解大部分弱口令。
3. **Pass-the-Hash (PtH)**：若无法破解，攻击者无需明文密码。利用 SMB 协议的特性，攻击者可直接使用 NTLM 哈希值进行身份验证（如使用 `psexec.py` 或 `wmiexec.py`），实现横向移动。这是 Windows 认证协议设计上的重大缺陷。

#### 2.3.2 Linux /etc/shadow 与 yescrypt
Linux 系统密码哈希存储在 `/etc/shadow` 文件中。现代发行版（如 Ubuntu 22.04+）已默认采用 **yescrypt** 算法（前缀 `$y$`），取代了传统的 SHA-512 crypt（前缀 `$6$`）。
**实战利用**：
使用 `unshadow` 合并 `/etc/passwd` 和 `/etc/shadow`，然后使用 Hashcat (`-m 1800` 或 `-m 22500`) 进行破解。对于 yescrypt，由于其内存硬特性，GPU 破解效率大幅降低，必须依赖高质量的定向字典。

#### 2.3.3 MySQL 认证协议与哈希提取
MySQL 5.7+ 默认使用 `caching_sha2_password`，而老版本使用 `mysql_native_password`。
**mysql_native_password**：`SHA1(SHA1(password))`。无 Salt，极易被彩虹表破解。
**caching_sha2_password**：基于 SHA-256 和 PBKDF2，包含 Salt 和迭代次数，安全性大幅提升。
**实战利用**：通过 SQL 注入或读取 `mysql.user` 表提取哈希。对于老版本哈希，可直接在线解密；对于新版本，需使用 Hashcat (`-m 7400` 或 `-m 22500`) 进行爆破。若无法破解，可尝试利用 MySQL 的 `LOAD DATA LOCAL INFILE` 或 UDF 提权获取系统权限。

## 第三章：对称加密的代数结构、分组模式与逆向分析

对称加密的核心在于**混淆**和**扩散**。现代分组密码主要分为 Feistel 网络（如 DES, SM4）和 SPN 网络（如 AES）。在 Web 和 APP 测试中，AES 及其国密对标算法 SM4 是最常见的数据保护手段。

### 3.1 AES (Rijndael) 与国密 SM4 的底层架构

#### 3.1.1 AES 的有限域代数分析
AES 基于 **SPN（替换-置换网络）**。其所有运算均在伽罗瓦域 $GF(2^8)$ 上进行，不可约多项式为 $m(x) = x^8 + x^4 + x^3 + x + 1$。
- **SubBytes (S盒)**：S盒是 AES 唯一的非线性组件，由 $GF(2^8)$ 上的乘法逆元和仿射变换构成。其设计保证了差分均匀度最大值不超过 4，抵抗差分密码分析。
- **MixColumns**：使用 MDS 矩阵进行乘法，保证最大分支数为 5，确保 4 轮以上的 AES 能够抵抗截断差分攻击。

#### 3.1.2 国密 SM4 算法架构与逆向特征
SM4 采用 32 轮非线性迭代结构，属于 **非平衡 Feistel 网络**。分组长度和密钥长度均为 128 位。
**逆向识别特征**：
在反编译代码（如 IDA Pro 或 Ghidra）或前端 JS 中，若发现以下特征，即可确认为 SM4：
1. 32 轮循环结构。
2. 系统参数 FK 数组：`0xa3b1bac6`, `0x56aa3350`, `0x677d9197`, `0xb27022dc`。
3. 固定参数 CK 数组（32个32位整数，如 `0x00070e15`, `0x1c232a31`）。
4. 核心 T 变换中包含特定的循环左移操作（`<<< 2`, `<<< 10`, `<<< 18`, `<<< 24`）。

### 3.2 分组模式的高级密码分析与实战漏洞

AES/SM4 是块加密算法（16字节/块），必须配合分组模式（Mode）和填充（Padding）才能加密任意长度数据。模式的选择直接决定了系统的安全性。

#### 3.2.1 ECB 模式与模式泄漏
ECB（电子密码本）模式将明文按 16 字节分块，每块独立使用相同密钥加密。
**致命缺陷**：相同的明文块必然产生相同的密文块。这导致数据的统计模式被完全保留（如著名的“Tux 企鹅”加密后仍能看出轮廓）。
**实战利用**：在 Web 测试中，若发现相同参数（如 `id=1`）每次加密后的密文完全一致，即可判定为 ECB 模式。攻击者可通过**密文块重排（Block Reordering）**或**块替换（Block Substitution）** 攻击，在不解密的情况下篡改数据。例如，将管理员 Cookie 中的权限标识密文块替换为普通用户的密文块。

#### 3.2.2 CBC 模式与 Padding Oracle Attack (POA)
CBC（密码分组链接）模式引入了初始化向量（IV），每个明文块在加密前与前一个密文块异或。这是目前 Web 应用中最常见的模式。
**Padding Oracle 攻击原理**：
当使用 CBC 配合 PKCS#7 填充时，若后端在解密失败（填充错误）和业务逻辑错误时返回**不同的错误码或响应时间**，攻击者即可利用此 Oracle 逐字节解密密文。
CBC 解密公式：$P_i = D_K(C_i) \oplus C_{i-1}$。设 $I_i = D_K(C_i)$ 为中间状态。
攻击者篡改前一个密文块 $C_{i-1}$ 的最后一个字节，遍历 256 种可能。当服务器返回“填充正确”时，说明解密后的明文最后一个字节为 `0x01`。由此可推导出 $I_i$ 的最后一个字节，进而计算出原始明文 $P_i$。通过递推，可解密整个 Block，甚至加密任意伪造的 Payload。
**历史 CVE**：ASP.NET MS10-070、Apache Shiro-550/721、Java JSF。
**防御**：必须在解密前进行 HMAC 校验（Encrypt-then-MAC），或统一返回恒定时间的通用错误信息。

#### 3.2.3 CTR/GCM 模式与 Nonce 重用灾难
CTR 和 GCM 模式将分组密码转化为流密码。其安全性**绝对依赖于 Nonce（或 IV）的唯一性**。
**Nonce 重用后果**：
若两次加密使用了相同的 Key 和 Nonce：$C_1 = P_1 \oplus E_K(Nonce)$，$C_2 = P_2 \oplus E_K(Nonce)$。
攻击者计算 $C_1 \oplus C_2 = P_1 \oplus P_2$。密钥流被完全抵消。若 $P_1$ 包含已知明文（如 HTTP 头部、JSON 格式），可通过 crib-dragging 技术瞬间恢复全部明文。
在 GCM 模式下，Nonce 重用还会导致 GHASH 认证密钥 $H$ 泄漏（Forbidden Attack），攻击者可伪造任意密文的 MAC 标签，彻底摧毁系统的完整性。

### 3.3 对称加密的逆向提取与流量解密实战

在 APP 或小程序测试中，若遇到 AES/SM4 加密的请求体，测试者必须提取密钥（Key）、偏移量（IV）、模式（Mode）和填充方式（Padding）。

#### 3.3.1 静态分析与常量搜索
在 APK 反编译（JADX）或前端 JS 源码中，全局搜索以下特征：
- **AES 常量**：S 盒数组（`0x63, 0x7c, 0x77...`）、Rcon 数组。
- **SM4 常量**：FK 和 CK 数组。
- **硬编码密钥**：16/24/32 字节的随机字符串，或 Base64 编码的密钥。
- **模式标识**：`AES/CBC/PKCS5Padding`、`SM4/ECB/NoPadding`。

#### 3.3.2 动态 Hook 与内存 Dump
若密钥经过复杂派生（如 KDF）或存储在 Native 层（SO 文件），静态分析将失效。此时需使用 **Frida** 进行动态插桩。
**Frida Hook 策略**：
1. **Hook Java 层**：拦截 `javax.crypto.Cipher.getInstance` 获取模式和填充；拦截 `SecretKeySpec` 和 `IvParameterSpec` 的构造函数，直接 Dump 出 Key 和 IV 的字节数组。
2. **Hook Native 层**：若应用使用 OpenSSL 或 BoringSSL，Hook `EVP_CipherInit_ex` 或 `AES_set_encrypt_key`，提取传入的密钥和 IV 指针，读取内存数据。
3. **内存搜索**：在应用运行时，使用 Frida 扫描内存堆，寻找已知的明文特征（如 `{"code":200}`），在其附近寻找可能作为 Key 或 IV 的高熵数据块。

## 第四章：非对称加密的数论基础、协议缺陷与侧信道分析

非对称加密的安全性建立在数论难题之上。RSA 基于大整数分解（IFP），ECC 基于椭圆曲线离散对数（ECDLP）。在 Web 登录、API 签名、TLS 握手中，非对称加密扮演着密钥交换与身份认证的核心角色。

### 4.1 RSA 的代数缺陷与常见攻击向量

RSA 的加解密公式为 $C = M^e \pmod n$ 和 $M = C^d \pmod n$。参数选择不当会导致多项式时间内的破解。

#### 4.1.1 小指数广播攻击 (Håstad's Broadcast Attack)
若使用极小的公钥指数（如 $e=3$），且同一明文 $M$ 被加密发送给 3 个不同的接收者（模数 $n_1, n_2, n_3$ 互素）：
$C_i = M^3 \pmod{n_i}$
根据**中国剩余定理（CRT）**，攻击者可以在整数域上计算出唯一的 $C' = M^3 \pmod{n_1 n_2 n_3}$。由于 $M < n_i$，必然有 $M^3 < n_1 n_2 n_3$。因此，$C'$ 就是 $M^3$ 的精确整数结果，直接对 $C'$ 开三次方根即可在 $O(1)$ 时间内恢复明文 $M$。

#### 4.1.2 Wiener's 攻击与 Boneh-Durfee 攻击 (小私钥)
为了加速解密（如智能卡环境），开发者可能选择极小的私钥 $d$。Wiener 证明了，若 $d < \frac{1}{3} n^{0.25}$，攻击者可通过对 $\frac{e}{n}$ 进行**连分数展开（Continued Fraction Expansion）**。由于 $ed - k\phi(n) = 1$，且 $\phi(n) \approx n$，可知 $\frac{e}{n} \approx \frac{k}{d}$。连分数展开的收敛子将直接暴露 $\frac{k}{d}$，从而恢复私钥 $d$。Boneh-Durfee 利用格基规约（LLL算法）将这一界限扩展到了 $d < n^{0.292}$。

#### 4.1.3 Coppersmith 小根攻击 (Stereotyped Messages)
若攻击者已知明文 $M$ 的大部分高位（例如，明文包含固定的 HTTP 头部，仅尾部随机），设 $M = M_{known} + x$，其中 $x$ 是未知的小整数。
加密方程为 $(M_{known} + x)^e \equiv C \pmod n$。
Coppersmith 证明了，若 $x < n^{1/e}$，可通过构造格（Lattice）并应用 LLL 算法，在多项式时间内求出小根 $x$。当 $e=3$ 时，若未知部分小于模数的 1/3，即可直接破解。

### 4.2 椭圆曲线密码学 (ECC) 与国密 SM2 的逆向陷阱

ECC 基于有限域上椭圆曲线点群的**离散对数问题（ECDLP）**。在相同安全强度下，ECC 的密钥长度远小于 RSA（256位 ECC ≈ 3072位 RSA）。

**SM2 的特殊性与逆向陷阱**：
SM2 并非标准的 ECDSA。其签名算法引入了**用户身份标识（Z值）** 和预计算哈希。
SM2 签名过程：
1. 计算 $M' = Z_A || M$（$Z_A$ 包含用户 ID、曲线参数、公钥坐标等）。
2. 计算 $e = SM3(M')$。
3. 生成随机数 $k$，计算点 $(x_1, y_1) = [k]G$。
4. 计算 $r = (e + x_1) \pmod n$。
5. 计算 $s = ((1+d_A)^{-1} \cdot (k - r \cdot d_A)) \pmod n$。
**逆向注意**：在逆向包含 SM2 的前端 JS 或 App 时，必须寻找 Z 值的生成逻辑。国密标准推荐默认的 User ID 为 `1234567812345678`（ASCII 编码）。若无法正确构造 $Z_A$，则无法伪造有效签名。此外，SM2 的加密算法引入了基于 KDF 的异或操作，密文结构通常为 `04 || C1(x,y) || C3(SM3 Hash) || C2(密文)`，这与 ECIES 标准有所不同。

### 4.3 侧信道攻击 (Side-Channel Attacks, SCA)

密码算法的数学完美性无法掩盖物理实现时的信息泄漏。
- **计时攻击 (Timing Attack)**：RSA 的模幂运算（Square-and-Multiply）执行时间取决于私钥中 '1' 的位数。若未采用 Montgomery 乘法或恒定时间实现，攻击者可通过测量解密耗时恢复私钥。
- **缓存时序攻击 (Cache-Timing Attack)**：AES 的 T-table 实现会因缓存命中/未命中产生时间差。攻击者可通过 **Flush+Reload** 攻击，监控 CPU L1/L2 Cache 的状态，精确推断出 S 盒的访问索引，从而在数千次加密内恢复完整 AES 密钥。

## 第五章：现代前端加密逆向、代码混淆与 Wasm 对抗

现代 Web 应用大量使用前端加密来对抗自动化脚本和中间人嗅探。前端代码虽在客户端执行，但通过高度混淆（Obfuscation）和 WebAssembly (Wasm) 编译，构建了极高的逆向壁垒。

### 5.1 控制流平坦化与 AST 反混淆

高级 JS 混淆器（如 Obfuscator.io, JScrambler）会应用**控制流平坦化（Control Flow Flattening）**。
**原理**：将原始代码块打乱，放入一个 `switch-case` 结构中，通过一个状态变量控制执行流。
```javascript
// 混淆后 (控制流平坦化)
let state = "1|2|0";
let parts = state.split("|");
let idx = 0;
while(true) {
  switch(parts[idx++]) {
    case "0": c = a + b; break;
    case "1": a = 1; break;
    case "2": b = 2; break;
  }
  if(idx >= parts.length) break;
}
```
**AST (抽象语法树) 还原方案**：
人工逆向此类代码是不现实的。必须编写 Babel 脚本，通过 AST 操作进行静态还原。
1. **解析**：使用 `@babel/parser` 将 JS 代码解析为 AST。
2. **定位**：遍历 AST，寻找包含 `SwitchStatement` 且被 `WhileStatement` 包裹的特征节点。
3. **提取控制流**：解析状态变量（如 `parts` 数组）的初始化赋值，获取真实的执行顺序。
4. **重组 CFG**：提取所有 `case` 块内部的 AST 节点，根据执行顺序将它们从 `switch` 结构中剥离，重新拼接为线性的 `BlockStatement`。
5. **死代码消除 (DCE)**：移除混淆器插入的无意义分支和冗余的字符串数组解密函数。

### 5.2 JSFuck 与极端编码对抗

**JSFuck** 是一种极端的 JS 混淆技术，仅使用 `[]()!+` 6 个字符即可编写并执行任意 JavaScript 代码。其原理是利用 JS 的隐式类型转换和 `eval` 构造器。
**实战案例**：在 CTF 或恶意脚本中，`alert(1)` 可能被编码为长达数万字符的 JSFuck 字符串。
**解密方法**：
1. **静态解码**：使用在线 JSFuck 解码器（如 jsfuck.com）直接还原。
2. **动态 Hook**：在浏览器 Console 中重写 `Function.prototype.constructor` 或 `eval`，在代码执行前拦截并打印还原后的明文源码。

### 5.3 WebAssembly (Wasm) 加密模块逆向

大厂的核心加密逻辑（如设备指纹生成、复杂滑块轨迹加密、私有协议签名）已全面迁移至 Wasm。Wasm 是一种基于栈的虚拟机字节码，直接逆向 `.wasm` 文件（如使用 Ghidra）极其困难，因为其丢失了高级控制流结构且充满间接调用。

**降维打击策略：动态插桩与边界 Hook**
不要试图反编译 Wasm 到 C/C++。Wasm 必须通过 JS 胶水代码与外部环境交互。我们只需 Hook JS 与 Wasm 的边界。
```javascript
// 拦截 Wasm 导出函数
const originalEncrypt = wasmInstance.exports.encrypt;
wasmInstance.exports.encrypt = function(ptr, len) {
    // 读取传入 Wasm 的明文内存 (假设 UTF-8 编码)
    const memory = new Uint8Array(wasmInstance.exports.memory.buffer);
    const plainText = new TextDecoder().decode(memory.subarray(ptr, ptr + len));
    console.log("[Hook] PlainText:", plainText);
    
    // 调用原始函数
    const resultPtr = originalEncrypt(ptr, len);
    
    // 读取 Wasm 返回的密文内存 (假设返回 32 字节哈希)
    const cipherText = memory.subarray(resultPtr, resultPtr + 32);
    console.log("[Hook] CipherText:", Array.from(cipherText).map(b => b.toString(16).padStart(2, '0')).join(''));
    
    return resultPtr;
};
```
通过这种方式，无论 Wasm 内部逻辑多复杂，我们都能直接获取输入输出，甚至直接将该 Wasm 模块打包为 Node.js 服务，作为我们的“加密黑盒”使用。

### 5.4 软件保护技术：加壳、VMP 与反编译对抗

在客户端安全测试（如 APK 逆向、EXE 漏洞挖掘）中，开发者常使用商业保护工具（如 VMProtect, Themida, 梆梆加固, 腾讯乐固）对二进制文件进行加固。
**保护机制**：
1. **代码虚拟化 (VMP)**：将原生 x86/ARM 指令转换为自定义的虚拟机字节码，运行时由内嵌的 VM 解释执行。这彻底破坏了控制流图（CFG），使 IDA Pro 的 F5 反编译完全失效。
2. **DEX 整体加固与函数抽取**：在 Android 中，将 DEX 文件加密，运行时解密到内存；或更高级的**函数级抽取（VMP for DEX）**，将核心方法的指令抽离，替换为 Native 层的 JNI 调用。
**对抗策略**：
- **内存 Dump**：对于整体加固，在应用运行时，通过 Frida Hook `DexFile` 或 `OpenMemory`，直接从内存中 Dump 出解密后的 DEX 文件（如使用 `Frida-DEX-Dump`）。
- **指令追踪**：对于 VMP，使用 Unicorn 等 CPU 模拟框架或 Intel Pin 进行动态指令追踪（DBI），通过记录执行轨迹还原算法逻辑。

## 第六章：密码学工程实现中的致命陷阱与测试方法论

在安全审计与代码审查中，算法本身的正确性往往不是问题，问题出在**密码学原语的错误组合与工程滥用**。

### 6.1 密钥派生函数 (KDF) 的滥用

**错误场景**：直接使用用户的密码（如 "password123"）作为 AES-256 的密钥，或仅使用简单的 `MD5(password)` 截断作为密钥。
**后果**：AES-256 的 256 位安全性瞬间降维至人类密码的熵（通常不到 40 位）。攻击者无需破解 AES，只需爆破低熵密码字典。
**正确实践**：必须使用 **HKDF** (HMAC-based Extract-and-Expand Key Derivation Function) 或 **PBKDF2/Argon2**。使用盐值将低熵输入提取为高强度的伪随机主密钥（PRK）。

### 6.2 随机数生成器 (RNG) 的熵枯竭与 Z3 求解

**错误场景**：在多线程服务器或高并发容器中，使用 `Math.random()`（JS）、`rand()`（C）或 `java.util.Random`（Java）生成 IV、Nonce 或 Salt。
**后果**：这些是线性同余生成器（LCG）或伪随机数生成器，其状态空间极小且完全可预测。LCG 的递推公式为 $X_{n+1} = (aX_n + c) \pmod m$。
**Z3 求解器实战**：攻击者只需收集少量的连续输出（如 6 个 32 位整数），即可利用微软的 Z3 定理证明器（SMT Solver）建立方程组，在毫秒级恢复 RNG 的内部种子（Seed），从而预测所有未来的“随机数”。
```python
# Z3 求解 LCG 种子示例
from z3 import *
s = Solver()
seed = BitVec('seed', 64)
a = 0x5DEECE66D
c = 0xB
m = (1 << 48) - 1

# 假设截获了连续的两个输出 (取高32位)
out1 = 0x12345678
out2 = 0x9ABCDEF0

state1 = (seed * a + c) & m
state2 = (state1 * a + c) & m

s.add(LShR(state1, 16) == out1)
s.add(LShR(state2, 16) == out2)

if s.check() == sat:
    print("Seed recovered:", s.model()[seed])
```

### 6.3 加密与认证的顺序 (Encrypt-then-MAC vs MAC-then-Encrypt)

**错误场景**：先计算明文的 MAC，然后将明文和 MAC 一起加密（MAC-then-Encrypt）。
**后果**：容易受到 Padding Oracle 攻击，因为解密器在验证 MAC 之前必须先解密并处理填充，泄漏了填充状态。
**正确实践**：**Encrypt-then-MAC (EtM)**。先对明文加密，然后对**密文**计算 HMAC。接收方先验证 HMAC，若 HMAC 无效则直接丢弃，绝不触发解密逻辑。现代 AEAD 算法（如 AES-GCM, ChaCha20-Poly1305）已在底层强制执行了 EtM 逻辑。

### 6.4 安全测试方法论：黑盒、白盒与“50%定律”

在面对加密系统时，安全测试者必须建立正确的实战心态与方法论。

#### 6.4.1 黑盒测试中的加密壁垒
在黑盒测试（无源码）中，若遇到强加密（如 AES-CBC 且密钥未知，或 RSA-2048），且无法通过前端 JS 提取密钥，测试者应果断放弃正面破解。此时的策略应转向：
1. **寻找逻辑漏洞**：如密码重置链接的 Token 是否可预测、验证码是否可复用、是否存在越权访问。
2. **降级攻击**：检查 API 是否支持旧版本协议，或是否允许通过参数（如 `encrypt=false`）关闭加密。
3. **重放与篡改**：若使用 ECB 模式，尝试密文块重排；若未校验时间戳，尝试重放攻击。

#### 6.4.2 白盒测试中的源码审计
在白盒测试（有源码）中，重点审查密码学工程实现：
1. **硬编码密钥**：搜索代码库中的 `key`, `secret`, `salt`, `AES`, `RSA` 等关键字。
2. **弱算法使用**：检查是否使用了 MD5/SHA1 进行密码存储，是否使用了 ECB 模式，是否使用了 `Math.random()` 生成 Token。
3. **KDF 与 RNG**：验证密钥派生过程是否使用了 HKDF/PBKDF2，随机数生成是否调用了操作系统的 CSPRNG。

#### 6.4.3 逆向工程的“50%定律”与务实态度
在真实的逆向分析中，并非所有加密都能被破解。商业级加固、自定义私有算法、以及密钥存储在安全硬件（如 TEE/SE）中的情况，往往超出了软件逆向的范畴。
**务实原则**：
- **有能力解，没能力放**：不要在一个无法突破的加密点上死磕。若耗费数小时仍无法提取 AES 密钥，应迅速切换攻击面（如转向社工、钓鱼或物理渗透）。
- **信息收集决定成功率**：解密的成功率取决于你掌握的信息量。通过 JS 泄露、配置文件泄露、备份文件泄露获取的 Key/IV/Salt，往往是破解加密的唯一钥匙。
- **工具链的熟练度**：熟练掌握 Burp Suite 的 Intruder 与 Piper、Frida 的 Hook 脚本编写、Hashcat 的规则（Rules）与掩码（Masks）攻击，是提升那“50%成功率”的核心保障。

## 结语：密码分析的本质与安全测试的终极奥义

密码分析学（Cryptanalysis）并非寻找数学公式的捷径，而是**寻找理想数学模型与粗糙工程实现之间的裂缝**。无论是 CBC 模式的 Padding Oracle，还是侧信道的计时泄漏，亦或是前端 JS 的 AST 混淆与 Wasm 的内存边界，所有成功的破解均源于开发者对密码学原语边界的误解与傲慢。

作为安全研究员，我们的任务不仅是识别 `0-9a-f` 的 MD5 或 Base64 的 `==`，而是要具备透视协议栈的能力：从有限域 $GF(2^8)$ 的代数结构，到 CPU 缓存的微观物理行为，再到浏览器 V8 引擎的 JIT 编译机制与 Wasm 的线性内存模型。

在实战中，加密与编码是防御者的盾牌，也是攻击者的迷宫。掌握传输格式的解析、密码存储的演进、对称与非对称算法的代数缺陷，以及现代代码混淆的对抗技术，方能在复杂的现代加密系统中，精准定位那致命的阿喀琉斯之踵，完成从“脚本小子”到“密码分析工程师”的终极蜕变。唯有将密码学理论与一线渗透实战深度熔炼，才能在红蓝对抗的无声战场上，做到“见密文如见明文，视混淆如无物”。
