# http&https详解

## 第一部分：HTTP协议基础

### 1.1 什么是HTTP协议

HTTP（Hyper Text Transfer Protocol，超文本传输协议）是互联网上应用最广泛的网络协议之一，是客户端与服务器之间通信的基础。当你在浏览器中输入一个网址并按下回车，浏览器就会构造一个HTTP请求发送给服务器；服务器收到请求后进行处理，然后通过HTTP响应将结果返回给浏览器。

HTTP协议是一个**无连接、无状态**的协议。无连接意味着每次请求都需要建立新的连接，服务器处理完请求并返回响应后即断开连接。无状态意味着服务器不会自动保存客户端的状态信息，每个请求都是独立的——服务器不会记得你之前是否访问过，也不会记得你的身份。这也是为什么我们需要Cookie和Session机制来维持用户登录状态。

HTTP基于TCP/IP协议栈，默认使用80端口（HTTPS使用443端口）。它采用客户端/服务端（C/S）架构模型，通过可靠的TCP连接来传递数据。

### 1.2 HTTP请求与响应的基本流程

一次完整的HTTP通信包含以下步骤：

1. **客户端构造请求**：浏览器根据用户操作（输入网址、点击链接、提交表单等）构造HTTP请求报文
2. **建立TCP连接**：客户端与服务器通过三次握手建立TCP连接
3. **发送请求**：客户端将HTTP请求报文通过TCP连接发送给服务器
4. **服务器处理**：服务器接收请求报文，解析后执行相应处理（查询数据库、读取文件等）
5. **返回响应**：服务器构造HTTP响应报文，通过TCP连接返回给客户端
6. **关闭连接**：服务器关闭TCP连接（或保持连接以复用，取决于Connection头部）

当加入代理（如Burp Suite抓包工具）后，流程变为：客户端 → 代理工具 → 服务器，代理工具可以在请求转发前进行拦截、查看和修改。

## 第二部分：HTTP请求报文结构

HTTP请求报文由**请求行（Request Line）** 、**请求头（Request Headers）** 、**空行（Blank Line）** 和**请求体（Request Body）** 四个部分组成。

### 2.1 请求行（Request Line）

请求行是HTTP请求报文的第一行，由三个部分组成，各部分用空格分隔：

```
GET /index.html?page=1 HTTP/1.1
```

**①请求方法（Method）** ：告诉服务器要执行什么操作，如GET、POST、PUT、DELETE等。最常见的请求方法是GET和POST。

**②请求URI（Request-URI）** ：标识要访问的资源路径，可以包含查询参数（Query String）。URI与请求头中的Host字段共同组成完整的请求URL。

**③HTTP版本（HTTP Version）** ：标识使用的HTTP协议版本，如HTTP/1.0、HTTP/1.1、HTTP/2.0。目前主流仍为HTTP/1.1。

### 2.2 请求头（Request Headers）

请求头位于请求行之后，包含若干个键值对（Header Field），格式为`字段名: 字段值`，每个头部字段占一行。请求头用于向服务器传递额外的请求信息，如客户端类型、可接受的数据格式、身份凭证等。

**常见的请求头字段及其作用：**

| 头部字段 | 作用 | 示例值 |
|----------|------|--------|
| **Host** | 指定请求的目标服务器域名和端口 | `Host: www.example.com` |
| **User-Agent** | 标识客户端的类型、操作系统和版本 | `User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36` |
| **Accept** | 告知服务器客户端能处理的媒体类型 | `Accept: text/html,application/json` |
| **Cookie** | 携带服务器之前设置的会话凭证 | `Cookie: sessionId=abc123` |
| **Referer** | 标识当前请求的来源页面URL | `Referer: https://www.google.com/` |
| **Content-Type** | 请求体的MIME类型（主要用于POST/PUT） | `Content-Type: application/json` |
| **Content-Length** | 请求体的字节长度 | `Content-Length: 1024` |
| **Connection** | 控制连接是否保持 | `Connection: keep-alive` |

### 2.3 空行（Blank Line）

请求头与请求体之间用一个空行分隔，由回车符（CR，`\r`）和换行符（LF，`\n`）组成。空行是HTTP报文结构的固定组成部分，用于标识头部结束和正文开始。

### 2.4 请求体（Request Body）

请求体承载请求的数据内容，位于空行之后。GET请求通常没有请求体；POST、PUT等请求的请求体包含提交的数据。

**常见请求体格式：**

- **application/x-www-form-urlencoded**：键值对格式，如`username=admin&password=123456`
- **multipart/form-data**：多部分格式，用于文件上传
- **application/json**：JSON格式，如`{"username":"admin","password":"123456"}`

## 第三部分：HTTP请求方法完全解析

HTTP定义了一组请求方法（HTTP动词），用于表明要对资源执行的操作。以下是各种请求方法的详细解析。

### 3.1 GET方法

GET是最常见的HTTP方法，用于从服务器**获取**资源。

**核心特征：**
- 请求参数附加在URL后面，用`?`分隔URL与参数，多个参数用`&`连接
- 参数长度受限（浏览器限制约2048字符，服务器限制各异）
- **没有请求体（Body）**
- 是**安全**的（Safe）：GET请求不应修改服务器状态
- 是**幂等**的（Idempotent）：多次执行相同GET请求，结果相同
- 可被缓存，会被浏览器历史记录保存

**典型场景：** 访问网页、搜索查询、获取用户信息、下载文件。

**示例：**
```
GET /users?id=1001 HTTP/1.1
Host: api.example.com
```

### 3.2 POST方法

POST用于向服务器**提交数据**，通常用于创建新资源或执行非幂等操作。

**核心特征：**
- 请求参数放在**请求体（Body）** 中，可传输大量数据
- 是**非安全**的（会修改服务器状态）
- 是**非幂等**的（多次执行可能产生不同结果，如多次提交订单会创建多个订单）
- 默认不可缓存，不会被浏览器历史记录保存

**典型场景：** 用户注册、登录认证、提交表单、上传文件、创建订单。

**示例：**
```
POST /users HTTP/1.1
Host: api.example.com
Content-Type: application/json
Content-Length: 52

{"name":"张三","email":"zhangsan@example.com","password":"123456"}
```

### 3.3 其他请求方法

| 方法 | 作用 | 幂等性 | 有无请求体 | 典型场景 |
|------|------|--------|------------|----------|
| **HEAD** | 与GET相同，但不返回响应体 | 是 | 无 | 检查资源是否存在、获取文件大小 |
| **PUT** | 全量更新或创建资源 | 是 | 有 | 完整更新用户信息 |
| **DELETE** | 删除指定资源 | 是 | 通常无 | 删除用户、删除订单 |
| **PATCH** | 部分更新资源 | 是 | 有 | 只修改用户邮箱 |
| **OPTIONS** | 查询服务器支持的方法 | 是 | 无 | CORS预检请求 |
| **CONNECT** | 建立隧道连接 | 否 | 有 | HTTPS代理隧道 |

### 3.4 GET与POST的核心区别

| 对比维度 | GET | POST |
|----------|-----|------|
| 参数位置 | URL中 | 请求体中 |
| 数据长度限制 | 有（URL长度限制） | 无（理论上无限） |
| 可见性 | 参数在URL中可见 | 参数在请求体中，相对隐蔽 |
| 缓存 | 可缓存 | 不可缓存 |
| 幂等性 | 是 | 否 |
| 安全性 | 安全（只读） | 非安全（可修改数据） |
| 书签/历史 | 可被保存 | 不可被保存 |
| 典型用途 | 获取数据 | 提交数据 |

**重要说明**：GET和POST的安全性都是相对的。GET的参数在URL中明文可见，更容易被泄露（如浏览器历史、服务器日志）；POST的参数虽然不在URL中，但如果没有HTTPS加密，同样可以被网络中间人截获。真正的安全传输依赖于HTTPS加密。

## 第四部分：HTTP响应报文与状态码

### 4.1 响应报文结构

HTTP响应报文由**状态行（Status Line）** 、**响应头（Response Headers）** 、**空行**和**响应体（Response Body）** 四部分组成。

**状态行格式**：`HTTP版本 状态码 原因短语`

```
HTTP/1.1 200 OK
```

### 4.2 HTTP状态码分类

HTTP状态码采用三位数字编码，通过首位数字划分为五大类：

| 分类 | 范围 | 含义 | 代表状态码 |
|------|------|------|------------|
| **1xx 信息性** | 100-199 | 请求已接收，继续处理 | 100 Continue |
| **2xx 成功** | 200-299 | 请求已成功处理 | 200 OK |
| **3xx 重定向** | 300-399 | 需要进一步操作以完成请求 | 301、302 |
| **4xx 客户端错误** | 400-499 | 请求包含错误或无法处理 | 403、404 |
| **5xx 服务器错误** | 500-599 | 服务器处理请求时出错 | 500 Internal Server Error |

### 4.3 常见状态码详解

**200 OK**：最常见的状态码，表示请求已成功处理，服务器返回了请求的资源。

**301 Moved Permanently**：永久重定向。请求的资源已被永久移动到新位置，客户端应使用新URL。搜索引擎会将权重转移到新URL。

**302 Found**：临时重定向。请求的资源临时移动到新位置，客户端应继续使用原URL。搜索引擎不会转移权重。

**304 Not Modified**：未修改。客户端可使用缓存版本，服务器无需返回资源内容。

**400 Bad Request**：错误请求。请求包含语法错误，服务器无法理解。

**401 Unauthorized**：未授权。请求需要用户认证，但未提供有效凭证。

**403 Forbidden**：禁止访问。服务器理解请求但拒绝执行。常见于目录浏览被禁用、IP被封锁等情况。

**404 Not Found**：未找到。请求的资源在服务器上不存在。不指明是临时性还是永久性。

**405 Method Not Allowed**：方法不允许。请求方法不被该资源支持。

**500 Internal Server Error**：服务器内部错误。服务器遇到意外情况，无法完成请求。

**502 Bad Gateway**：网关错误。作为网关或代理的服务器从上游服务器收到无效响应。

**503 Service Unavailable**：服务不可用。服务器过载或正在维护，暂时无法处理请求。

### 4.4 状态码在安全测试中的应用

通过分析HTTP响应状态码，可以判断文件或目录是否存在，这是**目录扫描和路径探测**的核心原理：

- **200**：文件存在且可访问
- **403**：文件夹存在，但目录浏览被禁用，且无默认索引文件
- **404**：文件或文件夹不存在
- **302/301**：可能存在跳转，需要进一步验证（可能是资源存在，也可能是容错跳转）
- **500**：服务器错误，无法确定资源是否存在

专业目录扫描工具（如dirsearch、gobuster、Burp Suite Intruder）正是通过发送大量路径请求并分析返回状态码来发现隐藏的后台路径和敏感文件。

## 第五部分：HTTP头部字段深度解析

### 5.1 Host

Host字段指定请求的目标服务器域名和端口。当多个网站托管在同一台服务器上时（虚拟主机），服务器通过Host头区分请求应路由到哪个网站。

**安全测试价值：Host头注入漏洞**。攻击者篡改Host头可导致：
- 密码重置功能被操纵，重置链接发送到攻击者控制的域
- Web缓存投毒，将恶意内容缓存给其他用户
- 绕过基于Host的访问控制

### 5.2 User-Agent

User-Agent标识客户端的类型、操作系统和版本。服务器通过UA头判断访问设备类型，返回对应的页面版本。

**实战应用**：
- 修改UA头可伪装成不同设备（手机/电脑/不同浏览器）
- 绕过仅限特定客户端访问的限制
- 测试服务器对不同客户端的响应差异

### 5.3 Cookie

Cookie是服务器通过`Set-Cookie`响应头发送给客户端的小型文本数据，存储在客户端。客户端在后续请求中通过`Cookie`请求头将数据传回服务器。

**Cookie的安全风险**：
- 存储在客户端，用户可查看、修改、删除
- 可能被XSS攻击窃取
- 可能被篡改导致越权访问

**安全测试价值**：通过对比登录前后的Cookie变化，可以识别身份凭证字段。将登录后的Cookie复制到未登录的请求中，可能绕过登录直接访问后台——这是**越权访问漏洞**的典型测试方法。

### 5.4 Referer

Referer提供用户来源页面的URL。服务器可通过Referer判断请求来源，常用于防盗链和访问控制。

**安全测试价值**：Referer头可能泄露敏感信息（如包含Token的URL）。同时，缺少或伪造Referer可能绕过某些基于Referer的访问控制。

### 5.5 Content-Type与Content-Length

- **Content-Type**：请求体或响应体的MIME类型，如`application/json`、`text/html`、`image/png`
- **Content-Length**：请求体或响应体的字节长度

## 第六部分：HTTPS协议与TLS握手

### 6.1 HTTP与HTTPS的核心区别

| 对比维度 | HTTP | HTTPS |
|----------|------|-------|
| 数据传输 | **明文传输**，可被窃听 | **加密传输**，保证机密性 |
| 默认端口 | 80 | 443 |
| 连接建立 | TCP三次握手后即可传输 | TCP三次握手 + TLS握手 |
| 安全性 | 无身份认证，可能被中间人攻击 | 提供机密性、完整性和身份认证 |
| 证书 | 不需要 | 需要CA签发的数字证书 |
| 性能 | 较快 | 略慢（加解密开销） |

### 6.2 HTTPS的加密机制

HTTPS使用三种密码学技术的组合：

**对称加密**：使用同一个密钥进行加密和解密，速度快，适合大量数据传输。HTTPS在数据传输阶段使用对称加密（如AES）。

**非对称加密**：使用公钥加密、私钥解密（或反之），速度慢但安全性高。HTTPS在证书验证和密钥协商阶段使用非对称加密（如RSA、ECDHE）。

**哈希算法**：将任意长度的信息转换为固定长度的值，算法不可逆，用于保证数据完整性（如SHA-256）。

### 6.3 TLS握手过程详解

TLS握手是HTTPS中最复杂的部分，其核心目标是在不安全的网络中安全地协商出对称加密密钥。握手过程分为以下阶段：

**阶段一：算法与参数协商**

1. **Client Hello**：客户端向服务器发送连接请求，包含：
   - 支持的TLS版本（如TLS 1.2、TLS 1.3）
   - 支持的加密套件列表（如`TLS_ECDHE_RSA_WITH_AES_256_GCM_SHA384`）
   - 一个随机数（Client Random）

2. **Server Hello**：服务器从客户端列表中选择双方都支持的加密套件和TLS版本，生成一个随机数（Server Random），返回给客户端。

**阶段二：身份认证与密钥交换**

3. **Certificate**：服务器将数字证书发送给客户端。证书包含服务器的公钥、域名、有效期和CA签名等信息。

4. **Server Key Exchange**（可选）：发送密钥交换所需的额外参数（如ECDHE的临时公钥）。

5. **Certificate Request**（可选）：如果服务器要求验证客户端身份，发送此消息。

6. **Server Hello Done**：服务器通知客户端Hello阶段结束。

**阶段三：密钥协商与握手完成**

7. **客户端验证证书**：客户端验证证书是否合法：
   - 证书是否由受信任的CA签发
   - 证书是否过期
   - 证书的域名是否与访问域名一致
   - 证书是否被吊销

8. **Client Key Exchange**：客户端生成一个随机数（Pre-Master Secret），用服务器公钥加密后发送给服务器。

9. **Change Cipher Spec**：客户端通知服务器，后续通信将使用协商的会话密钥加密。

10. **Finished**：客户端发送第一条加密消息，验证加密通道是否正常工作。

11. **服务器Change Cipher Spec + Finished**：服务器同样通知客户端并发送加密验证消息。

握手完成后，双方使用协商的会话密钥进行对称加密通信。

### 6.4 抓包工具对HTTPS的中间人攻击原理

Burp Suite、Charles、Fiddler等抓包工具能够解密HTTPS流量，本质上是执行了一次**主动中间人攻击（MITM）** 。

**工作原理**：

1. 抓包工具在首次启动时生成一张**自签名的根CA证书**
2. 用户手动将这张根证书安装到操作系统或浏览器的"**受信任根证书颁发机构**"存储区
3. 客户端发起HTTPS请求时，抓包工具拦截TLS握手
4. 工具与真实服务器完成TLS握手，获取服务器的真实证书
5. 工具提取真实证书中的域名信息，用根CA私钥**动态签发一张伪造证书**
6. 工具将伪造证书返回给客户端
7. 客户端验证证书链时，因根CA受信任而通过验证
8. 建立两条独立的TLS通道：客户端↔工具、工具↔真实服务器
9. **明文数据在工具内部完全可见**

**关键前提**：根证书必须安装到"受信任根证书颁发机构"存储区。如果安装到"个人"证书库或其他位置，TLS握手将失败。

**SSL Pinning（证书钉扎）** ：部分APP实现了SSL Pinning，在客户端代码中硬编码了服务器的证书或公钥指纹。当收到伪造证书时，客户端会校验证书是否与硬编码值匹配，不匹配则直接断开连接。这是对抗MITM抓包的常见防护手段。

## 第七部分：HTTP协议在安全测试中的应用

### 7.1 数据包的唯一性与可修改性

在安全测试中，部分漏洞**必须使用特定数据包才能复现**。直接通过URL访问会采用工具或浏览器的默认请求头，而数据包注入会保留原始请求头信息。

**典型场景**：
- 漏洞存在于APP的特定功能中，浏览器直接访问无法触发
- 服务器通过UA头等字段区分客户端类型，不同客户端返回不同内容
- 某些功能只在特定客户端中可用，缺少特定头部字段会导致访问失败或重定向

**安全测试启示**：当发现某漏洞在浏览器中无法复现时，应尝试抓取APP或小程序的原始请求数据包，在测试工具中完整复现数据包格式，重点保持Cookie、UA等关键字段一致。

### 7.2 Cookie篡改与越权测试

Cookie是HTTP头部中用于判断用户身份的关键字段。通过对比登录前后的数据包，可以观察到Cookie的变化：

- 未登录状态下访问后台页面返回"没有权限"提示（401或403）
- 登录后Cookie字段会新增身份凭证信息（如`username=xxx`、`sessionid=xxx`）
- 将登录后的Cookie复制到未登录的请求中，可能绕过登录直接访问后台

**安全测试价值**：修改Cookie字段可以测试系统的身份验证机制是否健全。如果服务器仅依赖Cookie中的特定字段判断用户身份且未做严格校验，就存在**越权访问漏洞**。

### 7.3 Host头注入测试

Host头注入漏洞允许攻击者篡改Host头，导致服务器执行非预期操作。

**测试方法**：在请求的Host头中填入另一个域名（如`attacker.com`），观察服务器如何处理。

**可能的影响**：
- 密码重置功能被操纵，重置链接指向攻击者控制的域
- Web缓存投毒，恶意内容被缓存给其他用户
- 绕过基于Host头的访问控制
- 服务器将请求分发到错误的虚拟主机

### 7.4 User-Agent与设备伪装

通过修改User-Agent头，可以模拟不同设备和浏览器访问目标网站。

**安全测试应用**：
- 测试服务器对不同客户端的响应差异
- 绕过仅限移动端访问的限制
- 发现仅在特定客户端中存在的漏洞

### 7.5 目录扫描与状态码判断

通过修改请求路径并分析返回状态码，可以判断文件或目录是否存在：

- **200**：文件存在
- **403**：文件夹存在但无索引文件
- **404**：文件或文件夹不存在
- **302/301**：可能存在跳转，需进一步验证
- **500**：服务器错误，无法确定

专业扫描工具（dirsearch、gobuster、Burp Suite Intruder）通过加载路径字典，依次尝试访问不同路径并分析返回状态码来发现隐藏路径。

## 第八部分：实用工具详解

### 8.1 Burp Suite

Burp Suite是Web安全测试的集成平台，核心功能包括：

**Proxy（代理）** ：在浏览器与目标服务器之间充当中间人，允许查看和修改所有通过流量。配置代理（127.0.0.1:8080）后，拦截请求数据包进行分析。

**Repeater（重放器）** ：捕获请求后发送到Repeater，修改参数后重新发送到服务器，分析响应变化。

**Intruder（入侵者）** ：自动化攻击测试，支持字典爆破、参数模糊测试等。

**Decoder（解码器）** ：对数据进行编码/解码，如URL编码、Base64、HTML实体等。

**Comparer（比较器）** ：对比两个数据包的差异，便于分析。

### 8.2 Postman

Postman是专业的API测试工具，支持从零构建完整数据包。

**核心功能**：
- 新建HTTP请求，设置请求地址、请求方法（GET/POST/PUT/DELETE等）
- 自定义请求头部（Headers），包括User-Agent、Cookie、Authorization等
- 支持多种请求体格式（form-data、x-www-form-urlencoded、JSON、XML等）
- 显示返回包内容，支持预览/源代码/美化代码三种查看方式
- 支持Cookie自动管理、批量测试、环境变量
- 支持授权操作（Basic Auth、Bearer Token、OAuth 2.0等）

**与Burp Suite的协同**：Postman提供灵活的API请求构建能力，Burp Suite擅长流量拦截与深度分析。二者可通过代理协同工作：将Postman的代理指向Burp（127.0.0.1:8080），实现动态篡改和深度分析。

**使用建议**：
- Postman更适合纯API测试和接口调试场景
- Burp Suite更适合安全测试和漏洞挖掘场景
- 二者协同使用可发挥最大效能

## 第九部分：知识速查总表

| 知识点 | 核心内容 | 重点/易混淆点 | 安全测试价值 |
|--------|----------|--------------|--------------|
| **HTTP请求报文结构** | 请求行+请求头+空行+请求体 | GET无请求体，POST有请求体 | 构造/修改数据包进行测试 |
| **GET vs POST** | GET获取资源，POST提交数据 | GET参数在URL，POST参数在Body | 参数注入测试 |
| **状态码分类** | 2xx成功、3xx重定向、4xx客户端错误、5xx服务器错误 | 403禁止vs404不存在；301永久vs302临时 | 目录扫描、漏洞探测 |
| **Cookie** | 客户端存储的身份凭证 | 可被查看、修改、窃取 | 越权访问测试 |
| **Session** | 服务端存储的用户状态 | 数据在服务端更安全 | 会话固定测试 |
| **HTTPS** | HTTP+SSL/TLS加密传输 | 默认端口443，需证书 | 证书校验绕过 |
| **TLS握手** | 证书验证+密钥协商+对称加密传输 | 非对称加密验证证书，对称加密传输数据 | 中间人攻击原理 |
| **Host头** | 指定请求的服务器域名 | 虚拟主机路由依据 | Host头注入测试 |
| **User-Agent** | 标识客户端类型和版本 | 服务器据此返回不同内容 | 设备伪装、绕过限制 |
| **Referer** | 请求来源页面URL | 防盗链、访问控制 | 信息泄露、绕过控制 |
| **Content-Type** | 请求体/响应体的MIME类型 | 决定数据解析方式 | 注入攻击测试 |
| **Burp Suite** | Web安全测试集成平台 | 代理+重放+爆破 | 抓包、改包、自动化测试 |
| **Postman** | API测试工具 | 从零构建请求 | 接口测试、API渗透 |
