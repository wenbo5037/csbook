# Windows 网络与目录服务（Active Directory）

## 1. AD DS 核心组件：从逻辑到物理的深度解构

Active Directory 并非单一的数据库，而是一个由**逻辑容器**、**物理拓扑**与**操作主机角色**交织而成的分布式系统。理解这三者的相互作用是解决复杂 AD 问题的前提。

### 1.1 逻辑结构：域、树、林与 OU 的本质

#### 1.1.1 安全边界 vs 管理边界
*   **林 (Forest)**: AD 的**唯一真实安全边界**。林内的所有域共享同一个 Schema、Configuration 分区和 Global Catalog。这意味着：
    *   林管理员（Enterprise Admins）对所有域拥有完全控制权。
    *   SID History、UPN Suffix、Schema 扩展在林范围内生效。
    *   **攻击面**: 攻陷林内任意一个域的 Domain Admins，理论上可通过 KRBTGT 哈希或跨域信任提升至 Enterprise Admins。**多域不等于多安全边界**。
*   **域 (Domain)**: **复制边界**与**策略应用边界**。每个域有独立的 NTDS.DIT 文件、独立的 KRBTGT 账户、独立的默认密码策略（Fine-Grained Password Policy 除外）。
*   **组织单元 (OU)**: 纯粹的**管理容器**。用于委派管理权限、链接 GPO、组织对象。OU 不参与复制拓扑计算，不影响安全边界。

#### 1.1.2 分区 (Naming Contexts) 与复制范围
AD 数据库 (NTDS.DIT) 在逻辑上分为四个主要分区，其复制行为截然不同：

| 分区名称 | DN 示例 | 复制范围 | 内容描述 | 关键注意事项 |
| :--- | :--- | :--- | :--- | :--- |
| **Schema** | CN=Schema,CN=Configuration,DC=forest | 全林 | 类定义、属性定义、OID | 修改需 Schema Admins；不可逆；必须先注册 Schema Master |
| **Configuration** | CN=Configuration,DC=forest | 全林 | 站点、子网、服务连接点、显示规范 | 应用程序（如 Exchange）依赖此分区；变更频繁可能导致复制风暴 |
| **Domain** | DC=domain,DC=com | 单域 | 用户、组、计算机、GPO | 日常运维主战场；PDC Emulator 对此分区有特殊权威 |
| **Application** | DC=DomainDnsZones / ForestDnsZones | 域/林 | DNS 区域数据 | 仅存储集成的 DNS 区域；非 DNS 相关对象不应存于此 |

> **⚠️ 工程警示**：在大型林中，Configuration 分区的变更会触发全林复制。若在业务高峰期批量修改站点链接或发布大量 SCP，可能导致 WAN 链路拥塞。建议使用 `repadmin /queue` 监控复制队列深度。

### 1.2 FSMO 五大操作主机：角色、影响与应急转移

FSMO (Flexible Single Master Operations) 是为了解决多主复制冲突而设计的**伪单主**机制。理解每个角色的“不可替代性”是制定灾备计划的关键。

#### 1.2.1 角色详解与故障影响分析

| 角色 | 范围 | 核心职责 | 离线即时影响 | 离线长期影响 | 推荐抢占时机 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **Schema Master** | 林 | 批准 Schema 更新 | 无法扩展 Schema；现有认证/复制正常 | 阻塞新应用部署；Exchange/CU 升级失败 | 确认原主机永久丢失且急需扩展 Schema |
| **Domain Naming Master** | 林 | 添加/删除域；创建 APP 分区 | 无法加/删域；现有域运行正常 | 阻塞林结构调整；DNS 区域创建失败 | 同上 |
| **PDC Emulator** | 域 | 时间源；密码首选写入；GPO 编辑锁；DFS-N | 密码更改延迟生效；时间漂移；GPO 编辑冲突 | 认证失败（时间偏差）；账户锁定异常 | **尽快**；该角色对用户体验影响最大 |
| **RID Master** | 域 | 分配 RID 池给 DC | 当前 DC 用完本地 RID 池后无法创建安全主体 | 全域无法新建用户/组/计算机 | 当多个 DC 报告 RID 耗尽时立即抢占 |
| **Infrastructure Master** | 域 | 维护跨域对象引用（Phantom Objects） | 跨域组成员显示为 SID 而非名称 | 跨域权限解析错误；Universal Group 缓存过期 | 仅在多域环境中重要；单域环境无影响 |

#### 1.2.2 转移 (Transfer) vs 抢占 (Seize)

*   **Graceful Transfer**: 原角色持有者在线，通过 RPC 协商移交。数据一致性有保障。
    ```powershell
    Move-ADDirectoryServiceOperationMasterRole -Identity TargetDC -OperationMasterRole PDCEmulator,RIDMaster
    ```
*   **Forceful Seize**: 原角色持有者永久离线。**强制修改元数据**。
    *   **风险**: 若原主机日后重新上线，会导致 USN 冲突、RID 重复、Schema 不一致等灾难性问题。
    *   **前置条件**: 必须确保原主机**永远不会**再次连接到网络（物理销毁或强制降级）。
    *   **命令**: `Move-ADDirectoryServiceOperationMasterRole ... -Force`
    *   **事后清理**: 使用 `ntdsutil metadata cleanup` 清除残留对象；检查 DNS SRV 记录是否更新。

#### 1.2.3 PDC Emulator 的特殊地位
尽管微软淡化了“主域控制器”概念，但 PDCe 依然是事实上的核心：
1.  **时间权威**: 林根域的 PDCe 是全林的 Stratum 1 时间源。所有其他 DC 向其同步。
2.  **密码复制优先级**: 密码更改首先复制到 PDCe。当用户在非 PDCe 上认证失败时，DC 会将请求转发给 PDCe 重试（防止因复制延迟导致的误拒）。
3.  **账户锁定**: 锁定状态由 PDCe 最终裁定。
4.  **GPO 版本控制**: GPMC 默认连接 PDCe 编辑 GPO，避免并发写入冲突。

### 1.3 全局编录 (GC) 与通用组成员资格缓存

#### 1.3.1 GC 的核心功能
GC 存储了林内**所有域**的所有对象的**部分副本**（Partial Attribute Set, PAS）。
*   **通用组展开**: 登录时，DC 查询 GC 获取用户的 Universal Group 成员资格。若 GC 不可用且未启用缓存，用户可能无法登录（除非是 Domain Admins 或缓存凭据）。
*   **跨域搜索**: Outlook 地址簿查找、打印机定位等依赖 GC。
*   **UPN 解析**: 当用户使用 UPN 登录时，本地 DC 可能不知道该 UPN 属于哪个域，需查询 GC 进行路由。

#### 1.3.2 Universal Group Membership Caching (UGMC)
在分支站点部署 GC 成本高（带宽/存储）。UGMC 允许本地 DC 缓存用户的通用组成员资格。
*   **机制**: 首次登录后，DC 将组成员资格缓存到本地 NTDS.DIT。后续登录直接使用缓存。
*   **刷新**: 每 8 小时向 GC 验证一次缓存有效性。
*   **限制**: 不适用于首次登录、UPN 路由、跨域搜索。仅加速已知用户的重复登录。
*   **配置**: Site 级别启用，粒度优于部署 GC。

### 1.4 站点拓扑与复制机制

#### 1.4.1 KCC 与 ISTG
*   **KCC (Knowledge Consistency Checker)**: 运行在每个 DC 上的进程，每 15 分钟评估一次，自动生成连接对象 (Connection Objects)。
*   **ISTG (Inter-Site Topology Generator)**: 每个站点选出一个 DC 作为 ISTG，负责计算**站间**复制拓扑。ISTG 不生成连接对象，而是生成站点链接桥接。

#### 1.4.2 复制协议选择

| 场景 | 协议 | 特点 | 调度 |
| :--- | :--- | :--- | :--- |
| **站内 (Intra-site)** | RPC over IP (TCP 135 + High Ports) | 实时通知；压缩；无调度限制 | 变更通知 (15s delay) + 拉取 |
| **站间 (Inter-site)** | RPC over IP 或 SMTP | 可调度；高度压缩；异步 | 基于 Schedule (默认 180min) |

> **💡 最佳实践**：除非有极端低带宽链路（<64kbps），否则**始终使用 RPC over IP**。SMTP 复制不支持 Domain 分区复制，仅支持 Schema/Config/GC，且已被微软标记为过时。

#### 1.4.3 USN 与复制冲突解决
*   **Update Sequence Number (USN)**: 每个 DC 维护本地递增计数器。每次修改分配一个新 USN。
*   **High-Watermark Vector**: DC 记录每个复制伙伴的最新 USN。复制时只请求 > HWM 的变更。
*   **Up-to-Dateness Vector**: 记录每个原始写入 DC 的最高 USN。防止循环复制。
*   **冲突解决**: 当两个 DC 修改同一属性时，采用**版本号 (Version Number)** 优先；版本相同则比较**时间戳**；时间戳相同则比较**Server GUID**。这与“最后写入胜出”不同，版本号增加的操作总是优先于仅时间戳更新的操作。

## 2. 核心网络服务：AD 的生命维持系统

AD 不是孤立运行的。DNS、DHCP、NTP 构成了其赖以生存的三角支撑。任何一个环节的故障都会导致 AD 功能退化甚至瘫痪。

### 2.1 DNS：AD 的命脉

#### 2.1.1 SRV 记录与客户端定位
AD 客户端不使用静态 IP 列表寻找 DC，而是通过 DNS SRV 记录动态发现。

```text
_ldap._tcp.dc._msdcs.<ForestRootDomain>      # 查找任意 DC
_ldap._tcp.<SiteName>._sites.dc._msdcs.<Domain> # 查找本站点 DC
_gc._tcp.<ForestRootDomain>                  # 查找 GC
_kerberos._tcp.dc._msdcs.<Domain>            # 查找 KDC
_pdc._tcp.<Domain>                           # 查找 PDC Emulator
```

**客户端定位算法**:
1.  查询 `_ldap._tcp.<Site>._sites.dc._msdcs.<Domain>`。
2.  若无结果（站点无 DC 或未正确映射子网），回退到 `_ldap._tcp.dc._msdcs.<Domain>`。
3.  收到多个 SRV 记录后，按 Priority 排序；同 Priority 按 Weight 加权随机选择。
4.  向选中的 DC 发送 LDAP Ping (CLDAP) 确认存活及站点归属。
5.  若 DC 返回“非最优站点”，客户端缓存该信息并尝试查找更优 DC。

#### 2.1.2 AD 集成区域 vs 传统区域

| 特性 | AD Integrated Zone | Standard Primary/Secondary |
| :--- | :--- | :--- |
| 存储位置 | NTDS.DIT (Application Partition) | 文本文件 (.dns) |
| 复制方式 | AD Replication (Multi-master) | Zone Transfer (AXFR/IXFR) |
| 安全性 | ACL on AD objects; Secure Updates only | IP-based transfer restrictions |
| 容错性 | 所有 DC 都是主服务器 | 单点故障 (Primary) |
| 记录大小限制 | 受 AD object size limit (~8MB) | 文件系统限制 |

> **⚠️ 关键配置**：务必启用 **Secure Dynamic Updates**。否则，任何加入域的机器都可以覆盖其他机器的 A/PTR 记录，导致中间人攻击或名称劫持。检查 DNS 控制台 → Zone Properties → Dynamic updates: "Secure only"。

#### 2.1.3 DNS 老化与清理 (Scavenging)
动态注册的记录若不清理，会导致僵尸记录堆积，引发解析错误或安全风险。
*   **No-Refresh Interval**: 记录创建后的一段时间内，不接受刷新（减少 AD 写入）。默认 7 天。
*   **Refresh Interval**: No-Refresh 结束后，接受刷新的窗口期。默认 7 天。
*   **Scavenging Period**: 超过 No-Refresh + Refresh 仍未刷新的记录被删除。
*   **公式**: 记录存活最短时间 = No-Refresh + Refresh = 14 天。
*   **陷阱**: 必须在**服务器级别**和**区域级别**同时启用 Scavenging。仅在区域启用无效。

### 2.2 DHCP 与 AD 的交互

#### 2.2.1 DHCP 授权机制
为防止流氓 DHCP 服务器，AD 环境中的 DHCP 服务器必须经过授权。
*   **检测流程**: DHCP 服务启动时，查询 `CN=NetServices,CN=Services,CN=Configuration,DC=forest` 下的 `dhcpClass` 对象。
*   **授权列表**: 只有列在此处的服务器才能绑定端口 67/68。
*   **例外**: 独立工作组服务器无法被授权，也无法在域环境中合法运行（会被域 DC 检测到并发送 DHCPINFORM 告知其停止服务）。

#### 2.2.2 DNS 动态更新代理
DHCP 服务器常代表客户端执行 DNS 更新（尤其是非 Windows 客户端或不支持动态更新的设备）。
*   **DnsUpdateProxy 组**: 若 DHCP 服务器在此组中，它创建的 DNS 记录**没有安全描述符**，允许任何认证用户后续接管更新。
*   **风险**: 若普通 DHCP 服务器加入此组，会导致大量不安全记录。
*   **最佳实践**: 仅在专用 DHCP 服务器上配置此组；或使用 DHCP Credential 指定专用服务账户执行更新。

### 2.3 NTP 时间同步：Kerberos 的隐形基石

#### 2.3.1 时间偏差容忍度
Kerberos 协议默认允许 **5 分钟** 时钟偏差。超过此值，AS_REQ/TGS_REQ 将被拒绝（Error Code: KRB_AP_ERR_SKEW）。
*   **注意**: 5 分钟是硬编码默认值，可通过注册表修改 `MaxClockSkew`，但**强烈不建议**。增大偏差会降低重放攻击防护能力。

#### 2.3.2 Windows 时间服务层级架构

```text
[External NTP Source (Stratum 1)]
          ^
          | (UDP 123)
          v
[Lin Root PDC Emulator (Stratum 2)] <-- 全林时间权威
          ^
          | (UDP 123 / RPC)
          v
[Other DCs in Forest Root (Stratum 3)]
          ^
          |
          v
[Child Domain PDCe (Stratum 3/4)] --> [Child Domain DCs]
          ^
          |
          v
[Member Servers & Clients]
```

#### 2.3.3 配置与排错命令

```powershell
# 查看时间源与层级
w32tm /query /status

# 强制重新同步
w32tm /resync /force

# 追踪时间同步详细日志
w32tm /debug /enable /file:C:\Temp\w32time.log /size:10000000 /entries:0-300

# 检查外部 NTP 可达性
w32tm /stripchart /computer:pool.ntp.org /samples:5

# 重置为默认域层次结构
net stop w32time
w32tm /unregister
w32tm /register
net start w32time
w32tm /config /syncfromflags:domhier /update
```

> **⚠️ 虚拟化陷阱**: 域控制器**绝不能**启用 Hyper-V Time Synchronization Integration Service。虚拟机应从 AD 层次结构同步时间，而非宿主机。否则，当宿主机时间漂移或暂停恢复时，会导致 DC 时间跳变，触发 USN Rollback Protection 或 Kerberos 故障。

## 3. 混合身份与云集成：迈向零信任的桥梁

现代企业不再是纯本地环境。Entra ID（Azure AD）与本地 AD 的融合是当前架构的主流。理解两者的差异与同步机制是避免“身份分裂”的关键。

### 3.1 本地 AD vs Entra ID：范式转变

| 维度 | Active Directory (On-Prem) | Entra ID (Cloud) |
| :--- | :--- | :--- |
| 协议 | LDAP, Kerberos, NTLM, RPC | REST, OAuth2, OIDC, SAML |
| 结构 | OU, GPO, Forest/Domain | Flat Tenant, Groups, Administrative Units |
| 认证 | 密码哈希, 智能卡, Certificate | MFA, FIDO2, Conditional Access, Token |
| 设备管理 | GPO, SCCM | Intune (MDM/MAM), Autopilot |
| 身份锚点 | SID, sAMAccountName | ObjectId (GUID), UPN |
| 扩展性 | Schema Extension (重量级) | Directory Extensions / Graph API (轻量级) |

### 3.2 Azure AD Connect (AAD Connect) 同步引擎

#### 3.2.1 同步架构
AAD Connect 不是简单的 LDAP 导出工具，而是一个**声明式规则引擎**。
*   **Source Anchor (ImmutableID)**: 连接本地对象与云对象的纽带。默认为 `objectGUID` 的 Base64 编码。**切勿更改**，否则会导致云账户重建、邮箱丢失。
*   **Sync Rules**: 内置规则定义了属性流（Attribute Flow）、Join 逻辑、Projection 规则。自定义规则应使用 Advanced Editor，避免直接修改内置规则。
*   **Staging Mode**: 生产环境必须有一台 Staging Server。当 Active Server 故障时，可快速切换。Staging Server 持续同步但不导出，确保热备数据新鲜。

#### 3.2.2 三种认证模式对比

| 模式 | 原理 | 优点 | 缺点 | 适用场景 |
| :--- | :--- | :--- | :--- | :--- |
| **Password Hash Sync (PHS)** | 同步 NTLM Hash 的派生值到云端。云端独立验证。 | 简单可靠；云认证不依赖本地；支持泄露检测 | Hash 上云（虽经处理）；不支持某些遗留认证 | 大多数企业首选 |
| **Pass-Through Auth (PTA)** | 云端将认证请求转发给本地 Agent 验证。 | Hash 不上云；实时验证本地策略 | 依赖本地 Agent 可用性；需部署多个 Agent HA | 合规要求 Hash 不出境 |
| **Federation (AD FS)** | 云端信任本地 STS 签发的 Token。 | 支持复杂声明转换；智能卡/CBA | 架构复杂；高可用成本高；单点故障风险大 | 已有 AD FS 投资；特殊认证需求 |

> **💡 趋势建议**：除非有明确的合规或技术障碍，否则**优先选择 PHS**。微软的安全功能（如 Identity Protection、Leaked Credentials Detection）依赖云端 Hash 比对。PTA 和 Federation 正在逐步被边缘化。

### 3.3 混合加入 (Hybrid Join) 与设备管理

#### 3.3.1 Hybrid Azure AD Join 流程
1.  设备加入本地 AD。
2.  AAD Connect 同步设备对象到 Entra ID（仅同步 Enabled 计算机）。
3.  设备登录时，Windows 检测到云中存在对应对象，自动注册设备并获取 PRT (Primary Refresh Token)。
4.  **关键点**: 设备必须在 Entra ID 中处于 **Pending** 状态，直到设备完成自注册。若 AAD Connect 同步了但设备未注册，检查 SCP (Service Connection Point) 配置。

#### 3.3.2 Intune 与 GPO 共存策略
*   **Co-management**: SCCM + Intune 并行管理。工作负载可逐项迁移（如 Compliance, Windows Update, Endpoint Protection）。
*   **GPO vs Intune 冲突**: Intune MDM 策略通常**优先于** GPO。但若 GPO 设置了“阻止 MDM 策略”的注册表项，则 GPO 胜出。
*   **Group Policy Analytics**: Intune 提供工具扫描现有 GPO，评估哪些可迁移至 MDM，哪些不支持。

### 3.4 条件访问 (Conditional Access) 与零信任

条件访问是 Entra ID 的策略引擎，替代了传统的基于网络的防火墙规则。
*   **信号**: 用户身份、设备合规性、位置、应用、风险等级。
*   **决策**: Block, Grant (Require MFA/Compliant Device/Hybrid Join), Session Controls。
*   **PRT 的作用**: PRT 是设备级别的令牌，使 SSO 和条件访问设备合规性检查成为可能。若 PRT 丢失或无效，用户将被视为“未知设备”，触发更严格的 CA 策略。
*   **排错**: 使用 Entra ID Sign-in Logs → Conditional Access Tab 查看具体哪条策略命中、原因及详细信息。

## 4. 网络排错体系：从命令行到数据包的系统化方法

AD 问题本质上是网络问题。掌握分层排错工具链是高级工程师的核心竞争力。

### 4.1 Windows 防火墙高级安全 (WFAS) 规则链

WFAS 是有状态的包过滤器，规则处理顺序严格：

```text
[Inbound Packet]
      |
      v
1. Allow rules with Edge Traversal? (If applicable)
      |
      v
2. Authenticated Bypass rules? (IPsec exemptions)
      |
      v
3. Allow rules matching packet? --> ALLOW
      |
      v
4. Block rules matching packet? --> BLOCK
      |
      v
5. Default Action (Block for Inbound, Allow for Outbound)
```

**关键排错点**:
*   **Rule Precedence**: Allow 优先于 Block（Authenticated Bypass 除外）。这与 Linux iptables 的“先匹配先生效”不同。
*   **Profile Matching**: 规则必须匹配当前网络配置文件（Domain/Private/Public）。若 DC 被识别为 Public Network（常见于 NIC 断开重连），所有 Domain Profile 规则失效。
*   **Logging**: 默认关闭。排错时必须开启：`Set-NetFirewallProfile -All -LogAllowed True -LogBlocked True -LogFileName C:\Temp\FW.log`。

### 4.3 Wireshark 深度协议分析

#### 4.3.1 Kerberos 故障解码
| Error Code | 含义 | 常见原因 |
| :--- | :--- | :--- |
| KDC_ERR_C_PRINCIPAL_UNKNOWN (6) | 客户端主体未知 | 用户名拼写错误；账户禁用/删除；SPN 缺失 |
| KDC_ERR_S_PRINCIPAL_UNKNOWN (7) | 服务主体未知 | SPN 未注册；SPN 重复；DNS 解析错误 |
| KRB_AP_ERR_SKEW (37) | 时钟偏差过大 | NTP 故障；VM 时间同步冲突 |
| KDC_ERR_PREAUTH_FAILED (24) | 预认证失败 | 密码错误；RC4 被禁用但客户端不支持 AES |
| KDC_ERR_BADOPTION (13) | 无效选项 | 委派配置错误；TGT 转发标志被拒 |

**过滤技巧**:
```wireshark
kerberos.error_code != 0                    # 仅显示错误
kerberos.msg_type == 10                     # AS-REQ
kerberos.CNameString contains "admin"       # 特定用户
kerberos.sname.name_string contains "HTTP"  # 特定服务
```

#### 4.3.2 LDAP 性能与绑定分析
*   **Bind Request/Response**: 检查 Simple Bind vs SASL。Simple Bind 明文传输密码（除非 LDAPS）。
*   **Search Request**: 关注 `baseObject`, `scope`, `filter`, `attributes`。
    *   **Subtree Search on Root**: 性能杀手。应限定 Base DN。
    *   **Unindexed Attributes**: 导致服务器遍历整个 DB。检查 DC 事件日志 1644。
*   **Paged Results Control**: 大数据集查询必须分页。单次返回 >1000 条可能被截断。

#### 4.3.3 SMB/RPC 排错
*   **SMB Negotiate**: 检查 Dialect 版本。SMB1 应被拒绝。
*   **RPC Bind**: 检查 Interface UUID。AD 使用多种 RPC 接口（DRSUAPI, SAMR, LSARPC）。
*   **DCERPC Fault**: Status code 指示拒绝原因（Access Denied, Unavailable 等）。

## 5. 进阶主题：安全加固与架构优化

### 5.1 AD 安全基线要点
*   **Protected Users Group**: 成员禁止 NTLM、CredSSP、Des 加密；TGT 有效期缩短至 4 小时；不可委派。适用于高价值账户。
*   **Authentication Policies & Silos**: 精细控制“谁可以从哪里以何种方式认证”。例如：Admin 只能从 PAW 工作站登录。
*   **LSA Protection (RunAsPPL)**: 防止 LSASS 内存转储。配合 Credential Guard 使用。
*   **LDAP Signing & Channel Binding**: 强制签名防止中继。2020 年后默认启用，但需提前审计兼容性问题。
*   **Disable SMBv1 / LLMNR / NetBIOS**: 消除经典攻击向量。

### 5.2 站点拓扑设计原则
*   **子网映射准确性**: 错误的子网-站点关联导致客户端跨站点认证/复制。定期审计 `CN=Subnets,CN=Sites,...`。
*   **Bridge All Site Links (BASL)**: 默认启用。假设网络是全路由的。若存在非路由隔离网络，**必须禁用 BASL** 并手动创建站点链接桥。
*   **复制频率调优**: 核心站点间 15-30 分钟；边缘站点 60-180 分钟。避免过度复制消耗带宽。
*   **RODC 部署**: 物理安全差的分支机构使用 RODC。密码复制策略 (PRP) 严格控制哪些账户密码可缓存。

### 5.3 监控与可观测性
*   **AD Health Checks**: 自动化脚本定期检查 DCDIAG、复制状态、DNS、时间同步。
*   **Event Log Forwarding**: 关键事件（4624, 4625, 4720, 4726, 5136, 1644）转发至 SIEM。
*   **Performance Counters**: 监控 `NTDS\LDAP Searches/sec`, `NTDS\DRA Pending Replication Operations`, `DNS\Query Received/sec`。
*   **Third-Party Tools**: PingCastle (安全评估), Purple Knight (漏洞扫描), ADRecon (枚举).

## 6. 纠错与勘误说明

在编写本文档过程中，进行了以下关键技术点的交叉验证与修正：

1.  **FSMO 角色数量**: 明确区分林级（2个）与域级（3个）角色。避免笼统说“5个FSMO”而不分范围。
2.  **GC 与登录依赖**: 修正了“GC 不可用就无法登录”的说法。准确表述为：仅在首次登录或通用组成员资格未缓存时受影响。Domain Admins 和本地缓存凭据不受限。
3.  **DNS Scavenging 默认值**: 确认 Windows Server 2016+ 默认 No-Refresh=7d, Refresh=7d。旧版文档可能引用不同值。强调必须双层启用。
4.  **AAD Connect ImmutableID**: 澄清了 mS-DS-ConsistencyGuid 作为新一代 Source Anchor 的趋势。对于新用户，AAD Connect 可自动使用此属性替代 objectGUID，提高灵活性。文中补充了此演进。
5.  **Kerberos AES 支持**: 确认 Windows Vista/Server 2008+ 支持 AES。但某些旧应用仍硬编码 RC4。启用 AES-only 前必须全面测试。
6.  **W32Time 虚拟化**: 重申 Hyper-V Time Sync 对 DC 的危害。这是生产环境中最常见的配置错误之一。
7.  **WFAS 规则优先级**: 再次确认 Allow > Block 的规则。这与许多工程师的直觉（Block 优先）相反，是排错时的常见盲点。
8.  **Hybrid Join SCP**: 明确 SCP 存在于 Configuration NC，而非 Domain NC。这是手动配置时容易出错的位置。
