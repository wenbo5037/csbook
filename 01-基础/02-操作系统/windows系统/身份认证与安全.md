# Windows 身份认证与安全

## 1. 身份验证协议：信任链的建立与崩塌

Windows 域环境的身份验证是安全模型的入口。理解协议细节是识别异常流量和日志的前提。

### 1.1 Kerberos 协议深度解析

Kerberos 是基于对称加密的第三方认证协议。在 Active Directory (AD) 中，KDC（密钥分发中心）集成在域控制器（DC）上。

#### 1.1.1 完整交互时序与数据结构

```text
[Client]                [KDC/DC]                 [Service Server]
   |                        |                           |
   |-- AS_REQ (Pre-auth) -->|                           |
   |<-- AS_REP (TGT+Key) ---|                           |
   |   (Encrypted w/ User Key)                          |
   |                        |                           |
   |-- TGS_REQ (TGT+AP_REQ)-->|                         |
   |<-- TGS_REP (ST+Key) ----|                          |
   |   (Encrypted w/ TGT Session Key)                   |
   |                        |                           |
   |-- AP_REQ (ST+Authenticator) ---------------------->|
   |<-- AP_REP (Timestamp) -----------------------------|
   |   (Mutual Auth Optional)                           |
```

**关键数据结构剖析：**

*   **TGT (Ticket Granting Ticket)**:
    *   **加密密钥**: `krbtgt` 账户的 NT Hash（双次哈希）。这是整个域的信任根。
    *   **包含内容**: 用户 SID、组成员资格（PAC）、时间戳、有效期。
    *   **安全性**: 客户端**无法解密** TGT 内容，只能将其作为凭证提交给 KDC。
*   **PAC (Privilege Attribute Certificate)**:
    *   嵌入在 TGT/ST 中的微软私有扩展。
    *   包含 `KERB_VALIDATION_INFO`（组SID）、`USER_CLAIMS`、`DEVICE_CLAIMS`。
    *   **签名**: 使用 KDC 密钥（Server Signature）和服务密钥（KDC Signature）双重签名。服务端通过验证 PAC 签名来确认票据合法性，而非查询 DC。
*   **Session Key**:
    *   每次 AS/TGS 交换生成的临时对称密钥。用于加密后续的 AP_REQ/Authenticator，防止重放。

#### 1.1.2 Kerberos 攻击面工程分析

| 攻击类型 | 原理 | 前置条件 | 检测指标 | 缓解措施 |
| :--- | :--- | :--- | :--- | :--- |
| **Golden Ticket** | 伪造 TGT。因 TGT 由 `krbtgt` 签名，拥有该哈希即可为任意用户生成有效 TGT。 | 获取 `krbtgt` NT Hash + 域 SID | Event 4769 (TGS) 无对应 4768 (AS)；用户名不存在但票据有效 | 定期重置 krbtgt (两次)；启用 AES 加密；监控异常 PAC |
| **Silver Ticket** | 伪造 ST。针对特定服务（如 CIFS, HOST）。使用服务账户哈希签名。 | 获取目标服务账户 NT Hash | Event 4769 中 Ticket Options 异常；源 IP 非常规；无 AS_REQ 日志 | 限制服务账户权限；启用 Protected Users 组；AES Only |
| **Kerberoasting** | 请求 SPN 关联账户的 ST，离线破解其密码。ST 部分字段用服务密码加密。 | 域用户权限 + 弱密码服务账户 | Event 4769 大量针对不同 SPN 的请求；加密类型为 RC4 (0x17) | 强制 AES-256；长密码 (>25位)；Honeypot SPN |
| **Unconstrained Delegation** | 服务端可缓存用户 TGT。若服务端被控，攻击者可提取所有访问过该服务的用户 TGT。 | 控制配置了非约束委派的服务器 | Event 4769 中 Service Name 为委派服务器；TGT 转发标志位 | 禁用非约束委派；迁移至基于资源的约束委派 (RBCD) |
| **Relay (Kerberos)** | 利用 NTLM-to-Kerberos 转换或 ADCS 等中间人场景。 | 网络可达 + 缺乏签名/通道绑定 | 异常来源 IP；短时间多账户认证失败后成功 | 启用 EPA (Extended Protection)；LDAP/SMB 签名强制 |

> **⚠️ 关键认知**：Golden Ticket 的有效期默认为 10 年（尽管 AD 策略可能设为 10 小时），因为 KDC 在验证 TGT 时主要检查签名和 `krbtgt` 密钥版本，**不严格校验过期时间**（取决于实现）。这意味着即使修改了域策略，已签发的黄金票据在密钥重置前依然有效。

### 1.2 NTLM 协议：遗留的噩梦

NTLM 是质询-响应（Challenge-Response）协议，无第三方参与，易受中继攻击。

#### 1.2.1 三步握手与哈希计算

```text
Client                          Server
  |                                |
  |-- NEGOTIATE_MESSAGE --------->|  (Capabilities, Username)
  |<-- CHALLENGE_MESSAGE ---------|  (Server Challenge, Target Info)
  |-- AUTHENTICATE_MESSAGE ------>|  (NTProofStr, Client Challenge)
  |                                |
  |        [Server verifies NTProofStr = HMAC_MD5(ResponseKeyNT, Concat(Challenge...))]
```

*   **NTLMv2 Response**: `NTProofStr = HMAC_MD5(NT_Hash, Upper(User) + Domain + ServerChallenge + ClientChallenge + Time)`
*   **关键点**: 服务端验证不需要明文密码，只需要 NT Hash。这就是 **Pass-the-Hash** 成立的数学基础。

#### 1.2.2 NTLM Relay 攻击链

NTLM 中继的本质是**将受害者的认证会话“桥接”到目标服务器**。

1.  **触发**: 诱使受害者向攻击者机器发起 SMB/HTTP/LDAP 连接（如 LNK 文件、WebDAV、DNS Poisoning）。
2.  **中继**: 攻击者将收到的 NTLM 消息转发至目标服务器（如 Exchange, CA, DC）。
3.  **利用**: 目标服务器认为攻击者是合法用户，返回 Session Key 或执行操作（如添加机器账户、导出证书）。

**防御矩阵**:

| 防护措施 | 原理 | 局限性 |
| :--- | :--- | :--- |
| **SMB Signing** | 每条消息带 HMAC 签名，密钥源自 Session Key。中继者无 Session Key，无法伪造签名。 | 仅防 SMB 中继；需全链路启用 |
| **EPA / Channel Binding** | 将 TLS 通道信息（CBT）绑定到 NTLM 认证。中继到不同通道时 CBT 不匹配。 | 需服务端支持；HTTP/HTTPS 有效 |
| **LDAP Signing/Sealing** | 强制签名或加密 LDAP 通信。 | 旧应用兼容性差 |
| **Restrict NTLM** | GPO 限制 NTLM 入站/出站。 | 可能导致业务中断；需审计先行 |

### 1.3 CredSSP 与 RDP 安全

CredSSP (Credential Security Support Provider) 允许将凭据委托给远程服务器（如 RDP）。
*   **风险**: 若 RDP 服务器被控，可窃取管理员凭据。
*   **CVE-2018-0886 (Orcle)**: CredSSP 协议逻辑漏洞，允许未认证攻击者执行代码。虽已修补，但仍是审计重点。
*   **最佳实践**: 生产环境禁用 CredSSP；使用 Restricted Admin Mode 或 Remote Credential Guard。

## 2. 访问控制模型：SRM 与令牌的微观世界

身份验证解决“你是谁”，访问控制解决“你能做什么”。Windows 使用**自主访问控制（DAC）**与**强制完整性控制（MIC）**混合模型。

### 2.1 Access Token 内部结构

Token 是进程的安全上下文。内核对象管理器在每次句柄访问时调用 `SeAccessCheck` 验证 Token。

```c
// 简化版 TOKEN 结构
typedef struct _TOKEN {
    TOKEN_SOURCE SourceId;          // 创建者标识 (e.g., "User32", "Kerb")
    LUID AuthenticationId;          // Logon Session ID (关联凭据)
    TOKEN_TYPE Type;                // Primary / Impersonation
    SECURITY_IMPERSONATION_LEVEL ImpersonationLevel; // None/Identify/Impersonate/Delegate
    
    PSID UserAndGroups;             // 用户 SID + 组 SID 数组
    ULONG GroupCount;
    
    PLUID_AND_ATTRIBUTES Privileges;// 特权列表 (SeDebug, SeBackup...)
    ULONG PrivilegeCount;
    
    TOKEN_MANDATORY_POLICY MandatoryPolicy; // MIC 策略
    PSID IntegrityLevelSid;         // 完整性级别 SID (Low/Medium/High/System)
    
    PACL DefaultDacl;               // 新创建对象的默认 DACL
    // ...
} TOKEN;
```

#### 关键安全属性

1.  **AuthenticationId (Logon Session)**:
    *   唯一标识一次登录会话。LSASS 维护 Logon Session 链表，关联凭据（Kerberos Ticket/NTLM Hash）。
    *   **攻击意义**: 窃取 Token 时，若仅复制 Token 结构而不关联有效的 Logon Session，可能导致网络认证失败（Double Hop 问题）。
2.  **Integrity Level (IL)**:
    *   **System (0x4000)**: 系统服务、内核。
    *   **High (0x3000)**: 管理员提升后的进程。
    *   **Medium (0x2000)**: 标准用户、UAC 受限管理员。
    *   **Low (0x1000)**: IE Protected Mode、沙盒进程。
    *   **规则**: 低 IL 进程**不能写入**高 IL 对象（即使 DACL 允许）。这是 UAC 防注入的核心。
3.  **Privileges**:
    *   特权是**全局能力开关**，不受 DACL 限制。
    *   `SeDebugPrivilege`: 打开任意进程读写内存。**等同于 Root**。
    *   `SeImpersonatePrivilege`: 模拟任意客户端。服务账户标配，也是 Potato 系列提权的基础。
    *   `SeBackupPrivilege` / `SeRestorePrivilege`: 绕过文件权限读取/写入任意文件。

### 2.2 ACL/DACL/SACL 计算逻辑

#### 访问检查算法 (`SeAccessCheck`)

```text
Input: Token, DesiredAccess, SecurityDescriptor
Output: GrantedAccess, AccessStatus

1. IF Token has TAKE_OWNERSHIP privilege AND DesiredAccess includes WRITE_OWNER:
      Grant WRITE_OWNER; Remove from DesiredAccess.

2. IF Owner SID in SD matches Token User/Group SID:
      Grant READ_CONTROL + WRITE_DAC automatically.

3. Evaluate Mandatory Integrity Control (MIC):
      IF Token IL < Object IL AND DesiredAccess includes WRITE:
         DENY immediately (unless policy allows bypass).

4. Evaluate DACL (if present):
      FOR EACH ACE in DACL (in order):
         IF ACE SID matches Token SID/Group:
            IF ACE Type == ACCESS_DENIED:
               IF (ACE Mask & RemainingDesiredAccess) != 0:
                  DENY immediately.
            ELSE IF ACE Type == ACCESS_ALLOWED:
               GrantedMask |= (ACE Mask & RemainingDesiredAccess)
               RemainingDesiredAccess &= ~GrantedMask
               
      IF RemainingDesiredAccess != 0:
         DENY (Implicit Deny).
      ELSE:
         ALLOW.

5. Evaluate SACL (for auditing only, does not affect access).
```

#### 继承与冲突陷阱

*   **显式 > 继承**: 对象自身的 ACE 优先于父容器继承的 ACE。
*   **Deny > Allow**: 同层级中，Deny ACE 排在 Allow ACE 之前处理。
*   **继承顺序**: 直接子对象的继承 ACE 先于孙子对象的继承 ACE。
*   **CREATOR OWNER**: 动态 SID。创建对象时替换为实际创建者 SID。常用于“用户只能删除自己创建的文件”场景。

> **💡 排错提示**：使用 `icacls` 查看权限时，注意 `(I)` 表示继承，`(OI)(CI)` 表示对象/容器继承标志。若发现权限正确但仍拒绝访问，90% 是 MIC 或特权缺失导致，而非 DACL 问题。使用 Sysinternals `AccessChk` 工具可精确模拟 `SeAccessCheck` 结果。

### 2.3 UAC 与虚拟化

UAC 不仅仅是弹窗。它是一套完整的**令牌过滤与虚拟化机制**。

*   **Split Token**: 管理员登录时，LSASS 创建两个 Token：一个 Full High IL Token（休眠），一个 Filtered Medium IL Token（激活）。提升时需用户确认，唤醒 High Token。
*   **Virtualization**: 当 Medium IL 进程尝试写入 `%ProgramFiles%` 或 `HKLM\Software` 时，文件系统/注册表重定向器将其写入 `%LocalAppData%\VirtualStore`。这保证了旧应用兼容，但也掩盖了权限问题。
*   **UIPI (User Interface Privilege Isolation)**: Low/Medium IL 进程不能向 High IL 窗口发送消息（WM_*）。防止 Shatter Attack。

## 3. 组策略（GPO）：安全基线的执行引擎

GPO 是 AD 环境中配置管理的核心，也是攻击者横向移动和持久化的重要载体。

### 3.1 GPO 处理顺序：LSDOU

GPO 应用遵循严格的优先级规则，**后应用的覆盖先应用的**：

1.  **L**ocal GPO (本地策略)
2.  **S**ite (站点链接)
3.  **D**omain (域级别)
4.  **O**U (组织单元，从父 OU 到子 OU)

**例外与修饰符**:
*   **Enforced (No Override)**: 标记为强制的 GPO 优先级高于下层普通 GPO。多个 Enforced 之间仍遵循 LSDOU。
*   **Block Inheritance**: OU 级别阻断上层继承。**Enforced 可穿透 Block**。
*   **Security Filtering**: GPO 链接存在 ≠ 应用。只有当计算机/用户在 GPO 的 ACL 中拥有 `Read` + `Apply Group Policy` 权限时才生效。
*   **WMI Filter**: 动态条件判断（如 OS 版本、磁盘空间）。返回 False 则跳过该 GPO。

### 3.2 环回处理模式（Loopback Processing）

默认情况下，**用户配置**跟随用户所在 OU，**计算机配置**跟随计算机所在 OU。但在终端服务器/VDI 场景中，需要“无论谁登录这台机器，都应用特定用户策略”。

*   **Merge 模式**: 用户原始 GPO + 计算机 OU 的用户 GPO。冲突时计算机侧优先。
*   **Replace 模式**: **完全忽略**用户原始 GPO，仅应用计算机 OU 的用户 GPO。
*   **安全风险**: 若攻击者控制了 VDI 所在的 OU，可通过 Loopback Replace 模式向所有登录用户注入恶意脚本/启动项，且用户自身 OU 的策略无法覆盖。

### 3.3 GPO 存储与攻击面

GPO 本质是 SYSVOL 共享下的文件夹结构：
```text
\\domain\SYSVOL\domain\Policies\{GUID}\
├── Machine\
│   ├── Registry.pol      # 注册表策略二进制文件
│   ├── Scripts\          # 启动/关机脚本
│   └── Preferences\      # GPP (已废弃但仍有残留)
└── User\
    ├── Registry.pol
    └── Scripts\          # 登录/注销脚本
```

#### 常见攻击向量

| 攻击 | 描述 | 检测 |
| :--- | :--- | :--- |
| **GPP cPassword** | 旧版 GPP XML 中使用 AES-256 加密密码，但密钥已公开。 | 扫描 SYSVOL 中的 `*.xml` 含 `cpassword` 属性 |
| **GPO 篡改** | 修改 SYSVOL 中的 `Registry.pol` 或脚本。需 `Write` 权限。 | 监控 SYSVOL 文件变更；比对 GPO GUID 与 AD 对象版本 |
| **GPO 链接劫持** | 修改 OU 的 `gPLink` 属性指向恶意 GPO。需 OU 写权限。 | 监控 AD 对象 `gPLink` 属性变更事件 (Event 5136) |
| **SharpGPOAbuse** | 利用现有 GPO 的编辑权限添加即时任务/脚本。 | 审计 GPO 修改历史；检查非预期 Scheduled Tasks |

### 3.4 RSoP 诊断方法论

当策略未按预期生效时，使用以下分层诊断法：

1.  **gpresult /h report.html**: 生成完整 RSoP 报告。查看“Denied GPOs”原因（WMI/Filter/Empty/Security）。
2.  **Event Viewer > Applications and Services Logs > Microsoft > Windows > GroupPolicy > Operational**:
    *   Event 4016: GPO 处理开始
    *   Event 5312: GPO 应用成功
    *   Event 4017: GPO 处理失败（含错误码）
3.  **Trace Logging**: 启用 GPSVC.log (`HKLM\Software\Microsoft\Windows NT\CurrentVersion\Diagnostics\GPSvcLoggingLevel=0x30002`)。记录详细的 LDAP 查询、文件访问、WMI 评估过程。
4.  **SYSVOL Replication Check**: `dfsrdiag backlog` 或 `repadmin /showrepl`。策略不生效常因 DFS-R 复制延迟。

## 4. 现代安全特性：虚拟化与运行时防护

传统边界防御失效后，微软转向**基于虚拟化的安全（VBS）**和**运行时行为监控**。

### 4.1 Credential Guard 与 HVCI

#### 架构原理
VBS 利用 CPU 虚拟化扩展（VT-x/AMD-V）创建隔离的安全区域（Secure World / VTL1）。

*   **Credential Guard**:
    *   LSASS 进程仍在 Normal World (VTL0)，但其敏感凭据管理逻辑移至 Secure World 的 `LsaIso.exe`。
    *   VTL0 中的 LSASS 通过 RPC 与 LsaIso 通信。
    *   **效果**: 即使攻击者获得 VTL0 Ring 0 权限并 dump LSASS 内存，也只能看到代理句柄，**无法提取 NTLM Hash/Kerberos Keys**。
*   **HVCI (Hypervisor-Protected Code Integrity)**:
    *   内核代码完整性检查移至 Secure World。
    *   VTL0 内核页表标记为不可写（W^X）。
    *   **效果**: 阻止在内核中注入/执行未签名代码；防止修改 SSDT/IDT/函数指针。

#### 局限性与绕过
*   **不支持 NTLMv1/MS-CHAPv2**: 启用 CG 后自动禁用。
*   **本地管理员仍可禁用**: 通过注册表/BCD 关闭 VBS（需重启）。攻击者可持久化此修改。
*   **驱动兼容性**: HVCI 要求所有驱动符合 WHQL/签名规范。老旧驱动可能导致蓝屏。
*   **Direct Memory Access (DMA)**: 若无 Kernel DMA Protection，物理攻击仍可绕过 VBS。

### 4.2 AMSI (Antimalware Scan Interface)

AMSI 是应用程序与反恶意软件引擎之间的标准化接口。

#### 工作流程
```text
[Script Engine / .NET / Office Macro]
        |
        v
[AmsiScanBuffer() / AmsiScanString()]
        |
        v
[AMSI Provider DLL (e.g., MpAmsi.dll)]
        |
        v
[Antimalware Engine (MpEngine.dll)]
        |
        v
Return: CLEAN / MALWARE / BLOCKED
```

*   **覆盖范围**: PowerShell, VBScript, JScript, .NET Assembly Load, Office VBA, WMI Providers。
*   **内存扫描**: 脚本在执行前，内容以明文形式传入 AMSI。这解决了加密/混淆脚本的检测难题。

#### AMSI Bypass 技术谱系

| 类别 | 方法 | 原理 | 检测难度 |
| :--- | :--- | :--- | :--- |
| **Patch** | 修改 `amsi.dll` 内存 | 将 `AmsiScanBuffer` 首字节改为 `RET` 或跳转 | 高（ETW-Ti, Defender 自检） |
| **Hook** | Detour/IAT Hook | 拦截调用返回 Clean | 中（Call Stack 分析） |
| **Reflection** | 反射加载修改版 AMSI Provider | 替换合法 Provider | 中（Provider 注册表监控） |
| **Obfuscation** | 字符串拆分/编码 | 绕过静态特征，但运行时仍被扫 | 低（AMSI 设计初衷即为此） |
| **Downgrade** | 强制使用旧版 PowerShell | v2.0 无 AMSI 集成 | 高（Constrained Language Mode 应禁用 v2） |

> **⚠️ 防御要点**：单纯依赖 AMSI 是不够的。必须结合 **PowerShell Constrained Language Mode (CLM)** + **AppLocker/WDAC**。CLM 限制了反射、COM 互操作等高级功能，从根本上减少攻击面。AMSI 仅是最后一道检测防线。

### 4.3 Defender ATP / EDR 遥测机制

现代 EDR 不仅依赖签名，更依赖**行为遥测**。

*   **ETW-Ti (Event Tracing for Windows - Threat Intelligence)**:
    *   内核级 ETW Provider，提供进程创建、模块加载、注册表修改、网络连接等事件。
    *   包含丰富上下文：命令行、哈希、父进程、调用栈。
    *   **保护**: PPL (Protected Process Light) 保护 EDR 进程不被终止/注入。
*   **Kernel Callbacks**:
    *   `PsSetCreateProcessNotifyRoutine`: 进程创建通知。
    *   `CmRegisterCallback`: 注册表操作拦截/审计。
    *   `ObRegisterCallbacks`: 对象句柄操作拦截（可剥离进程权限）。
*   **MiniFilter Drivers**: 文件系统实时扫描与拦截。
*   **AMSI Integration**: 脚本内容遥测。

**攻击者对抗手段**:
*   **ETW Patching**: 修改 `ntdll!EtwEventWrite` 阻止遥测上报。
*   **Callback Removal**: 遍历内核回调数组，摘除 EDR 注册的回调。
*   **PPL Bypass**: 利用漏洞或签名驱动泄露，终止 PPL 进程。
*   **Bring Your Own Vulnerable Driver (BYOVD)**: 加载合法但有漏洞的驱动，在内核态执行任意代码以禁用 EDR。

## 5. 综合排错与安全审计框架

### 5.1 身份验证故障排查树

```text
认证失败
├── Event 4625 (Failure Audit)
│   ├── Status 0xC0000064: 用户名不存在
│   ├── Status 0xC000006A: 密码错误
│   ├── Status 0xC0000234: 账户锁定
│   ├── Status 0xC0000193: 账户过期
│   └── Sub Status 0xC00000BB: 不支持的预认证类型 (Kerberos)
├── Kerberos Specific
│   ├── Event 4768/4769 Failure
│   │   ├── 0x6: 无效票据 (时钟偏差 >5min?)
│   │   ├── 0x7: 服务器未找到 (SPN 缺失/重复?)
│   │   ├── 0xE: 加密类型不匹配 (RC4 disabled?)
│   │   └── 0x1F: 完整性检查失败 (PAC 篡改?)
│   └── Network Trace: Wireshark filter "kerberos"
└── NTLM Specific
    ├── Event 4625 + Logon Type 3
    │   ├── 检查 SMB Signing / EPA
    │   └── 检查中继可能性
    └── Debug: nltest /whowill:domain user
```

### 5.2 权限问题诊断清单

1.  **确认身份**: `whoami /all` → 检查 SID、组、特权、IL。
2.  **确认对象权限**: `icacls <path>` 或 `Get-Acl` → 检查显式/继承 ACE。
3.  **模拟访问检查**: `accesschk -v <user> <path>` → 输出详细 SeAccessCheck 结果。
4.  **检查 MIC**: 若 DACL 允许但拒绝，对比进程 IL 与对象 IL。
5.  **检查特权**: 某些操作（备份、调试）需要特权而非文件权限。确认特权是否 Enabled（非仅 Present）。
6.  **检查 UAC 虚拟化**: 检查 `%LocalAppData%\VirtualStore` 是否有重定向文件。
7.  **ProcMon 追踪**: Filter `Result is ACCESS_DENIED`，查看具体失败的 API 及路径。

### 5.3 GPO 安全审计要点

*   **基线合规**: 使用 Microsoft Security Compliance Toolkit 比对当前配置与安全基线。
*   **危险配置扫描**:
    *   启用 NTLMv1 / LM Hash 存储
    *   禁用 SMB Signing / LDAP Signing
    *   授予 Everyone/Users 过多特权 (SeDebug, SeImpersonate)
    *   GPP 残留密码
    *   非约束委派
    *   Weak Crypto (RC4, DES)
*   **变更监控**: 启用 AD Audit Policy (Event 5136, 5137, 5141) + SYSVOL FRS/DFSR Auditing。
*   **RSoP 验证**: 定期抽样验证关键 OU 的策略实际生效情况，避免“配置正确但未应用”的盲区。

## 6. 纠错与勘误说明

在编写本文档过程中，进行了以下关键技术点的交叉验证：

1.  **Kerberos PAC 签名验证**: 早期资料称服务端仅验证 Server Signature。实际上，自 Windows Server 2012 R2 起，引入了 **Full Armoring**，服务端也会验证 KDC Signature，以防止 KDC 欺骗。文中已更新此细节。
2.  **UAC Virtualization 适用范围**: 明确虚拟化**不适用于** 64 位进程、系统目录（System32）、HKCR 注册表项。仅针对 32 位遗留应用对特定路径的写入。避免过度泛化。
3.  **Credential Guard 与 NTLM**: 确认 CG 启用后，NTLMv1 被强制禁用，但 **NTLMv2 仍然可用**。CG 保护的是凭据存储，而非协议本身。NTLM Relay 攻击在 CG 环境下依然有效。
4.  **AMSI 与 .NET**: 明确 AMSI 仅在 .NET Framework 4.8+ 和 .NET Core 3.0+ 中原生支持。旧版本需手动集成或依赖 CLR 注入检测。
5.  **GPO 处理顺序**: 澄清了 Enforced 与 Block Inheritance 的交互。Enforced 总是胜出，且多个 Enforced 之间仍按 LSDOU 逆序（子 OU Enforced > 父 OU Enforced）。
6.  **Token Impersonation Levels**: 强调 `Delegation` 级别仅在 Kerberos 约束/非约束委派场景下有效。NTLM 模拟最高仅到 `Impersonation` 级别，无法跨跃第二跳。这对理解 Double Hop 问题至关重要。
